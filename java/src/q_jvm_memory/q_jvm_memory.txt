
[ 자바 메모리 영역 ]
자바 프로그램이 시작되면 JVM 은 OS로부터 메모리를 할당받고, 해당 메모리를 여러 영역으로 나눈다.
이때 JVM의 메모리 영역 ( Runtime Data Area ) 는 크게 Heap 영역 , Stack 영역 , Method ( static ) 영역으로 나뉘며 데이터 타입에 따라 각 영역에 배치된다.

자바에서의 변수는 클래스변수, 인스턴스변수, 지역변수, 매개변수가 있으며, 변수의 선언 위치에 따라 변수 종류가 나뉜다.

클래스변수 : 클래스 영역 내에서 자료형 앞에 static 이 붙은 변수, 해당 클래스의 인스턴스들은 동일한 객체를 공유한다. 클래스가 메모리에 올라갈 때 생성된다.
인스턴스 변수 : 클래스 영역 내에서 자료형 앞에 static 이 붙지 않은 변수, 각 인스턴스들은 별도의 저장공간을 가진다. 인스턴스를 생성할 때 할당된다.
지역 변수 : 메서드 영역 내에서 생성되며, 메서드가 종료되면 소멸한다.
매개 변수 : 메서드 호출 시 전달 하는 값을 가지는 변수 ( 지역 변수처럼 메서드 종료 시 소멸 )

[ Method ( static ) 영역 - Metaspace 구현 환경 ]
Method 영역은 프로그램 시작부터 종료까지 존재한다.
JVM이 클래스를 로딩할 때 읽어들인 클래스 / 인터페이스에 대한 런타임 상수 풀, 멤버 변수, 클래스 변수,
상수, 생성자, 메서드 등에 대한 정보가 저장된다. ( 클래스에 대한 메타데이터 )
# 즉, 런타임 상수 풀, 클래스에 대한 메타데이터, 메서드, 메서드의 바이트 코드가 보관된다.

Method 영역에 적제된 데이터는 프로그램이 종료될 때 까지 남아있으며, 어디서든 접근할 수 있다.
클래스 변수 ( static ) 과 메서드는 무조건 Method 영역에 적재된다.
# 클래스 변수는 Method 영역에 존재하며, 모든 쓰레드가 접근이 가능하며, Method는 메타데이터 + 코드 ( 바이트 코드 ) 를 포함하고 있다.
# 즉, 메서드를 실행할 때 Method 영역에 있는 메서드 정보를 참고한다.
# 다만, 클래스 변수는 ( static ) java 8 부터는 Method 영역이 아닌 Heap 영역에 적제된다. 매우 중요!

클래스 내 일반 인스턴스 변수는 final 키워드가 붙어도 Method 영역이 아닌 Heap 으로 간다.

- 런타임 상수 풀 ( Runtime Constant Pool )
런타임 상수 풀에는 리터럴 상수와 심볼릭 참조가 들어간다.
심볼릭 참조는 이름 기반의 참조로 컴파일 시점에 이름을 보관하고 있다가, 런타임 시점에 링킹/해석 과정을 통해 실제 주소로 변환된다.
car.toString();
위의 코드를 컴파일 할 때 아래와 같은 정보가 런타임 상수 풀에 적재된다.
Car ( 클래스 ) , toString ( 메서드 ) , ()Ljava/Lang/String; ( 시그니처 )
# 여기서 시그니처는 메서드를 식별하기 위한 정보이다.
# 메서드는 오버로딩을 할 수 있기 때문에 식별을 위해 해당 구조로 표현된다.
# 시그니처는 (매개변수타입)반환타입 으로 구성이 된다.

이때 심볼릭 참조가 실제 주소로 변환되는 과정은 다음과 같다. ( 링킹/해석 과정 )
1. 클래스 파일이 클래스 로더에 의해 메모리에 올라갈 때 , 런타임 상수 풀에 메서드에 대한 정보를 이름 기반으로 적재한다. ( 클래스 , 메서드 , 시그니처 )
2. 런타임 시점에서 해당 메서드를 처음 호출하면, 해당 이름에 대한 정보를 기반으로 Method 영역을 참고하여 실제 메모리 주소로 변환한다. ( 이때 실제 메모리에 대한 정보가 캐싱된다. )
3. 이후 같은 메서드를 다시 호출하면 캐싱된 값을 통해 바로 해당 주소로 이동한다.
# 여기서 심볼릭 참조라는 별도의 공간이 있는 곳이 아니다. 메서드의 이름을 통해 실제 주소를 찾아간다는 의미에서 심볼릭 참조라고 하는 것이다.
# 컴파일 된 class 파일에는 이미 메서드에 대한 정보는 constant pool 에 올라가있다. 이것이 런타임 시점에 메모리에 로드되면 Runtime Constant Pool 이 된다.

[ Stack 영역 ]
Stack 영역은 메서드가 호출될 때 스택 프레임이 메모리에 쌓이며, 메서드가 종료될 때 해제된다.
각 스택 프레임은 기본형 타입의 지역 변수 및 매개 변수의 데이터와 중간 연산자 값, 반환 값이 저장 된다.
참조형 변수는 Heap 영역에 생성된 객체의 주소를 가지고 있다.

Stack 영역은 자료구조 Stack 처럼 후입선출 ( LIFO ) 의 구조를 가진다.
Stack 영역은 각 쓰레드마다 생성되기 때문에 동기화를 할 필요가 없다.

- Stack 프레임
하나의 메서드를 실행하기 위한 메모리 군집을 말한다.
한 메서드는 하나의 스택 프레임이 필요하며, 메서드가 호출되기 전 Stack 메모리에 스택 프레임을 생성한 후 메서드를 실행한다.
스택 프레임에는 Local Variables (지역 변수 + 매개변수), Operand Stack (연산용 스택), 리턴 값이 있으며 메서드가 종료되면 소멸된다.
# 연산용 스택은 사실 연산 중간 값이긴 하다.

[ Heap 영역 ]
런타임 시점에 new 연산자를 통해 동적으로 생성된 객체들이 저장되는 영역이다.
Heap 영역에는 실제 객체나 배열이 저장되는 공간이다.
# 이때 Method 영역에 있는 클래스의 메타데이터를 참고하여 객체를 생성한다.
Heap 영역은 JVM 과 생명 주기가 동일하며, 객체들은 주기적으로 생성/제거된다. ( 제거는 GC에 의해서만 제거된다. )

Heap 영역에 있는 실제 객체들의 메모리 주소를 가지고 있는 레퍼런스 변수는 Stack 영역에 존재한다.
# 만약 Stack s = new Stack() 으로 생성했다면, Stack stack 참조형 변수는 Stack 영역에 저장되고, new Stack() 인스턴스 변수는 Heap 영역에 적재된다.
이때 Stack 클래스의 안에 있는 각각의 멤버 변수들도 Heap 영역 안에 생성된다.

Heap 영역은 Stack 영역뿐만이 아니라 Method 영역에서도 접근할 수 있다. ( 클래스 변수 )
# 다만 Java 8 버전부터는 Heap 영역에 있는 실제 객체들을 Method 영역에서 참조하는 경우는 없다. ( static 변수가 Method 영역에서 Heap 영역으로 옮겨졌기 때문이다. )
또한 다른 Heap 영역에서도 접근할 수 있다. ( 객체 내부의 필드에서 접근할 수 있다. )
# static Car globalCar = new Car(); Method 영역에서 Heap 영역으로 접근하는 예제 ( Java 8 부터는 Heap 영역에서 Heap 영역으로 접근하는 예제 )

Heap 영역에 생성된 데이터는 가비지 컬렉터에 의해 제거된다.
즉, Stack 영역에 있는 레퍼런스 변수가 제거되고 GC Root 로 부터 도달 불가 상태가 되어, Heap 영역에 있는 객체의 주소를 아무도 가지지 않는다면 가비지 컬렉터에 의해 해제된다.

[ 정리 ]
Stack 영역은 쓰레드 갯수 마다 존재하지만 ( 쓰레드 마다 생성 ) , Heap 영역은 단 하나만 존재한다. ( 모든 쓰레드가 공유 )
이때 Stack 영역은 쓰레드가 다른 Stack 영역을 접근할 수 없다.
또한 Stack 영역의 생명주기는 매우 짧으며 ( 메서드의 시작부터 종료 ) , Heap 영역의 생명주기는 길다. ( 프로그램 시작부터 종료 )

그리고 생성된 객체는 Heap 영역에 존재하며, Stack 영역에서는 힙 공간에 있는 객체의 메모리 주소만 가지고 있다.
즉 Stack 영역은 기본형 타입 지역 변수와 참조 변수만 가지고 있다.

Stack 영역이 가득차면 java.lang.StackOverFlowError 오류를 발생시키고
Heap 영역이 가득차면 java.lang.OutOfMemoryError : Java Heap Space 오류를 발생시킨다.
# Heap 메모리의 크기는 -Xms 와 -Xmx 옵션을 통해 조절할 수 있다. ( -Xms : 초기 사이즈 , -Xmx : 최대 사이즈 )
그리고 Stack 영역은 Heap 영역보다 작지만, LIFO 라는 간단한 메모리 할당 기법을 사용하기 때문에 Heap 영역보다 빠르다.

[ Java 8 버전 부터의 static 위치 ]
위에서 클래스 변수 ( static ) 는 Method 영역에 위치한다고 했다.
Java 8 버전부터는 Heap 영역으로 이동되었다. 그 이유는 GC 대상으로 두며 permGen 문제를 해결하기 위해서이다. ( PermGen 의 OOM 방지 )
하지만 클래스 변수 ( static ) 는 GC Root 인 ClassLoader로 부터 도달 가능하기 때문에 클래스가 언로드되지 않는 한 사라지지 않는다.

클래스 변수는 클래스가 초기화 될 때 Heap 에 생성되고 기본 값으로 초기화된다. ( 만약 값을 설정하지 않았다면 0으로 초기화된다. )
# 물론 해당 클래스 변수의 값이 20으로 초기화 하려고 해도 우선 기본 값인 0 으로 초기화 한 후 20 으로 변경한다.

# 클래스가 언로드되는 상황은 ClassLoader 가 GC될 때에도 해당된다.
# static이 안 죽는 이유는 Heap이 아니라 ClassLoader가 살아있기 때문이다.

[ GC Root 의 종류 ]
1. Stack 영역 내 지역 변수 ( 모든 쓰레드 )
2. Static 필드 ( Class 객체 )
3. Thread 객체 ( 실행중인 쓰레드가 가지고 있는 객체 )
4. JNI ( C / C++ 이 참조하고 있는 객체 )
5. ClassLoader
# ClassLoader 는 Class 객체를 참조하고 있는 유일한 객체이기 때문에 GC Root가 되어 있다. 만약 ClassLoad가 없다면 Heap에 있는 Class 변수는 늘 GC 대상일 것이다.
