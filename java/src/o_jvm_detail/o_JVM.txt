[ JDK 와 JRE ]
JDK는 Java Development Kit 의 약자로 자바를 개발하는데 필요한 키트가 포함되어 있다.
여기서는 자바를 실행시키기 위한 Java Runtime Environment 도 함께 포함되어 있다.

JRE 는 Java Runtime Environment 의 약자로 자바를 실행시키기 위한 패키지를 가지고 있다.
개발이 아닌 실행만 시키기 위한 최소한의 패키지를 가지고 있다.

[ JVM 이란 ]
JVM 은 Java Virtual Machine 의 약자로, 기존 OS에 종속적인 문제를 해결하기 위한 개념이다.
JVM 을 사용하지 않는 C 언어는 각 운영체제 ( 리눅스 , 맥 , 윈도우 ) 에 맞게 별도로 컴파일을 해야하며, 환경에 따라 소스코드의 일부를 수정할 수도 있다.
JVM 은 이러한 단점을 개선시켜준다. JVM 위에서 동작하는 Java는 서로 다른 운영체제라 하더라도 소스코드를 수정하지 않아도 된다.

물론 JVM은 OS 별로 구현되기 때문에 OS에 맞는 JVM 을 설치할 수 있어야 한다.
JVM은 C 언어와 달리 컴파일과 변환을 수행하기 떄문에 C 언어보다 느리다는 단점을 가지고 있다. ( 물론 이는 JIT 컴파일러의 등장으로 C언어와의 격차는 줄어들었지만, 여전히 C언어가 더 빠르다. )
JVM위에 동작하는 JAVA 는 우선 .java 소스 코드를 .class 파일로 컴파일 하고 ( 바이트 코드 ) 이후 기계어인 바이너리 코드로 변환 하는 단계를 거치게 된다.
.java 소스 코드를 .class 파일로 컴파일 하는 것은 javac.exe 프로그램이 담당한다. ( 바이트 코드로 컴파일 - 자바 컴파일러 )
java.exe 프로그램은 JVM 을 실행시키며, 바이트 코드를 인터프리터 방식으로 실행시키거나, JIT 컴파일러를 통해 기계어인 바이너리 코드로 변환하여 실행한다. ( 기계어 변환 - JVM )
    ㄴ 기본적으로 인터프리터 방식이며, 성능 향상을 위해 일부는 JIT 컴파일로 최적화

자바 코드를 바이트 코드로 변환하는 것은 자바 컴파일러가 담당하며,
바이트 코드를 바이너리 코드로 변환 후 실제 운영체제에서 실행하는 것은 JVM 이 담당한다.
이때 JVM 은 각 운영체제에 맞는 바이너리 코드로 변환한다.

[ JVM 의 내부 구조 & 실행 구조 ]
JVM 은 클래스 로더를 통해 자바 어플리케이션을 읽고 자바 API와 함께 실행된다.
우선 자바 소스 파일이 어떻게 실행되는지 구성은 다음과 같다.

0. JVM 은 OS로 부터 메모리를 할당 받는다.
1. 자바 컴파일러가 자바 소스코드 .java 를 바이트 코드 ( .class ) 로 컴파일한다.
2. Class Loader 는 동적 로딩을 통해 필요한 클래스들을 로딩 및 링크하여 Runtime Data Area 에 올린다.
3. Runtime Data Area 에 로딩된 바이트 코드는 Execution Engine 을 통해 해석된다.

JVM 은 다음과 같이 구성되어 있다.
1. 클래스 로더
2. 실행 엔진 ( Execution Engine )
2.1 인터프리터
2.2 JIT 컴파일러
2.3 가비지 컬렉터
3. 런타임 데이터 영역 ( Runtime Data Area )
3.1 메서드 영역
3.2 힙 영역
3.3 PC Register
3.4 스택 영역
3.5 네이티브 메서드
4. JNI - 네이티브 메소드 인터페이스
5. 네이티브 메서드 라이브러리

[ 클래스 로더 ]
클래스 로더는 클래스 파일들을 JVM 내의 Runtime Data Area 영역에 배치한다.
이때 클래스 로더는 클래스 파일을 한번에 가져오지 않고, 어플리케이션이 필요할 때 동적으로 메모리에 적재한다.
클래스 로더는 Loading -> Linking -> Initialization 단계로 클래스 파일을 로딩한다.

Loading : 클래스 파일을 읽어와 JVM 내의 메모리에 적재한다.
Linking : 클래스 파일을 사용하기 위한 검증을 한다.
    - Verifying : 읽어들인 클래스가 JVM 명세대로 구성되어 있는지 판단한다.
    - Preparing : 클래스가 필요로 한 메모리를 할당한다.
    - Resolving : 클래스의 상수 풀 내 모든 심볼릭 레퍼런스를 다이렉트 레퍼런스로 변경한다.
Initialization : 클래스 변수들을 적절한 값으로 초기화한다.

[ 실행 엔진 ]
실행 엔진은 Runtime Data Area 에 있는 바이트 코드를 명령어 단위로 실행한다.
이때 실행 엔진은 JVM 내부에서 바이트 코드를 운영 체제가 이해할 수 있는 바이너리 코드로 변환한다.
# 바이트 코드는 JVM 만 이해할 수 있는 중간 레벨이다.
실행 엔진은 바이트 코드를 인터프리터 방식과 JIT 컴파일러 방식을 혼합하여 실행한다.

- 인터프리터 ( Interpreter )
바이트 코드의 명령을 하나씩 읽고 해석해서 바로 실행한다.

JVM 안에서는 기본적으로 인터프리터 방식으로 동작한다.
하지만 같은 명령어도 매번 해석을 수행하기 때문에 전체적인 속도가 느리다.

- JIT 컴파일러
인터프리터의 단점을 개선하기 위해 나온 개념으로, 자주 사용되는 바이트 코드를 네이티브 코드로 컴파일하여
이후 네이티브 코드를 실행하는 방식이다. ( 이때 해당 부분은 인터프리터 방식이 아닌 JIT 방식으로 사용 )
이때 네이티브 코드는 JVM 내부에 캐싱해두었다가 재사용된다.

속도는 인터프리터 보다 빠르지만, 네이티브 코드로 변경하는 비용이 소모되므로 모든 바이트 코드를 컴파일 하는 것이 아니라
자주 사용되는 바이트 코드부분에서 JIT 컴파일 방식을 활용한다.

- 가비지 컬렉터 ( GC )
JVM 은 가비지 컬렉터를 제공하여 Heap 메모리 영역에서 더이상 사용하지 않는 메모리를 자동으로 회수해준다.

GC는 자동으로 실행되지만, GC가 실행되는 시간은 정해져있지 않다.
특히 Full GC 가 발생하게 된다면 GC 를 제외한 모든 쓰레드가 멈춘다. 이는 장애를 발생시키거나, 서비스 응답 시간을 지연시킬 수 있다.
# System.gc() 를 통해 강제로 GC를 실행시킬 수 있지만, 실제 실행은 보장하지 않는다.

[ 런타임 데이터 영역 ( Runtime Data Area ) ]
런타임 데이터 영역은 JVM 메모리 영역으로 자바 어플리케이션을 실행할 때 사용되는 데이터들을 적재하는 영역이다.
이 구간은 크게 메서드 영역 / 힙 영역 / PC Register / 스택 영역 / 네이티브 메서드 로 나뉜다.
메서드 영역과 힙 영역은 모든 쓰레드가 공유하고 나머지는 각각의 쓰레드가 공유한다.

- 메서드 영역 ( 런타임 상수 풀 )
메서드 영역은 JVM이 시작될 때 생성되는 공간으로 바이트 코드를 메모리에 올릴 때 초기화되는 대상을 저장하기 위한 공간이다.
해당 영역은 JVM이 시작되고 프로그램이 종료될 때 까지 유지된다.

모든 쓰레드가 공유하는 영역이며 다음과 같은 정보들을 저장한다.
필드 정보 : 멤버 변수의 이름, 데이터 타입, 접근 제어자의 정보
메서드 정보 : 메소드 이름, return 타입, 함수 매개변수, 접근 제어자의 정보
타입 정보 : Class 인지 Interface 인지 여부 저장, Type의 속성, 이름 , Super Class의 이름

메서드 영역에는 정적 ( static ) 필드와 클래스 구조만을 가지고 있다.

런타임 상수 풀 : 메서드 영역 내에 별도로 관리되고 있는 영역으로, 각 클래스/인터페이스마다 별도의 테이블이 존재하며 클래스를 생성할 때 참조해야 할 정보들을
상수로 가지고 있다.
