자바가 C언어보다 느린 이유는 아키텍처 차이도 있지만 GC의 Stop-The-World 의 차이도 있다.
자바 어플리케이션의 성능이 나오지 않는다면, GC알고리즘을 변경하거나 GC튜닝이라는 절차도 활용될 수 있다.

하지만 실무에서는 GC 튜닝을 잘 사용하지 않는다. 그 이유는 효율이 좋지 않기 때문이며, 대부분 어플리케이션 코드를 개선시켜 메모리 최적화를 하는 것이 좋다.

[ GC의 주의사항 ]
1. GC 튜닝을 위해서 제공되는 옵션은 서비스마다 적정한 값이 있다.
이는 자바 어플리케이션의 특징마다 객체의 크기와 생존 기간에 차이가 있기 때문에 충분한 메모리 모니터링을 통해 적정한 값을 찾아나가야 한다.
다른 성공 사례를 따라 같은 옵션과 값을 제공하면 오히려 악영향을 줄 수 있다.

2. GC 튜닝은 가장 마지막에 하는 것이 좋다.
GC 튜닝을 하기 위해 필요한 지식, 튜닝의 리스크에 비해 얻는 이익은 그렇게 크진 않다.
따라서 GC 튜닝을 통해 어플리케이션 성능을 향상시키는 것 보다 소스 코드부분을 수정하여 어플리케이션을 개선 시키는 방향이 더 나을 수 있다.

GC가 발생하는 가장 큰 이유는 객체의 생성이 많기 때문이다. ( 물론 생명주기도 길어지면 GC 시간이 늘어난다. )
그렇기 때문에 소스 코드 내부에서 객체의 불필요한 생성을 줄이는 리팩토링 작업을 먼저 진행하는 것이 해결법이 될 수 있다.
예를 들어 String 보다 StringBuffer나 StringBuilder를 사용하고, 로그를 최대한 적게 쌓이게 해 메모리를 적게 사용하는 등의 리팩토링 하는 것이다.
이렇게 했음에도 성능이 개선되지 않으면 GC 튜닝을 진행하는 것이다.

[ GC 튜닝의 목표 ]
GC 튜닝은 GC에 대해 최적화 작업을 하는 것을 말한다. 그 중에서 Stop-The-World 시간이 긴 Full GC 관리이다.

1. Old 영역으로 넘어가는 객체 수를 최소화
Old 영역은 Young 영역에 비해 크기가 매우 크다. 이는 GC를 수행하는데 시간이 더 오래 걸린다는 뜻이다.
따라서 Young 영역에서 Old 영역으로 넘어가는 객체의 수를 최소화 하여 Full GC를 수행하는 횟수를 낮추는 것을 목표로 한다.

2. Full GC 시간 줄이기
Full GC의 수행 시간은 Minor GC보다 길기 때문에 Old 영역의 크기를 조절하는 것도 하나의 방법이다.
그렇다고 Old 영역의 크기를 너무 줄이면 Full GC 횟수가 늘어나거나 OutOfMemoryError 가 발생할 수 있고, Old 영역의 크기를 너무 늘리면 Full GC 수행 시간이 늘어날 수 있다.
이 사이의 중간 합의점을 잘 찾아야 한다.

[ GC 튜닝 과정 ]

1. jstat 명령어를 통해 해당 서비스의 GC 상태 모니터링
jstat -gcutil -t 8844 1000 0 명령을 통해 해당 JVM 어플리케이션의 메모리 상태를 확인할 수 있다. jstat 은 JDK 안에 포함된 모니터링 툴로 JDK 1.6 부터 사용할 수 있다.
jstat -gcutil -t [서비스의 PID번호] [추출 주기 (ms단위)] [출력 횟수 (0 이나 생략이면 무제한)]
-gcutil : 통계 요약을 백분위 ( % ) 단위로 보여주는 옵션
-t : Timestamp 출력 옵션으로 해당 옵션이 출력된 시간

jstat 에서 표시되는 정보는 다음과 같다.
S0 : Survivor0 영역의 사용율
S1 : Survivor1 영역의 사용율
E : Eden 영역의 사용율
O : Old 영역의 사용율
P : Permanent 영역의 사용률
YGC : Young 영역의 GC 발생 횟수
YGCT : Young 영역의 GC 시간
FGC : Full GC 발생 횟수
FGCT : Full GC 발생 시간
GCT : GC 총 시간

여기서 GC 발생 시간 / GC 발생 횟수 를 계산하면 각 GC 마다 평균적으로 얼마나 시간이 걸렸는지 알 수 있다.

2. 모니터링 결과 분석 후 GC 튜닝 여부 확인
GC의 주기 및 수행 시간을 기반으로 GC 튜닝을 할지 결정을 한다.

3. GC 알고리즘 선정
만약 GC 튜닝을 결정했다고 하면 어플리케이션의 특성 맞는 GC 옵션을 선택한다.
Parallel GC : 처리량이 중요한 시스템에서 사용
CMS GC : 응답시간이 중요한 시스템에서 사용
G1GC : 성능적으로 가장 우수한 GC 방식

4. 힙 메모리 크기 설정
GC의 발생 횟수와 수행 시간은 힙 메모리 크기와 연관이 있다. 힙의 최소크기 옵션과(-Xms) 힙 최대크기 옵션(-Xmx)의 크기를 적절히 설정한다.
그 외에 메모리 크기 설정옵션들은 다음의 것들이 있다.
-XX:NewRatio : Young 영역과 Old 영역의 비율
-XX:NewSize : Young 영역의 크기
-XX:SurvivorRatio : Eden 영역과 Survivor 영역의 비율

여기서 -Xms 과 -Xmx 옵션은 필수로 설정하길 권장하며, -XX:NewRatio 도 GC에 많은 영향을 준다.
-XX:NewRatio=1 로 설정하면 Young 과 Old 는 1:1로 비율이 설정된다. ( Heap 메모리가 1GB 라면 500MB : 500MB 가 된다. )
-XX:NewRatio=3 으로 설정하면 Young 과 Old 는 1:3 비율이 된다. ( Heap 메모리가 1GB 라면 250MB : 750MB 가 된다. )
즉 NewRatio 비율이 늘어날 수록 Young 의 크기는 줄어든다.

여기서 메모리의 크기가 커지면 GC의 횟수는 작아지지만, GC의 수행 시간은 길어지고,
메모리의 크기가 작아지면 GC의 횟수는 잦아지지만, GC의 수행 시간이 짧아지는 특징이 있다.

5. 튜닝 결과 분석
설정한 옵션에 따라 GC 상태를 확인 후 설정 값을 바꾸거나 어플리케이션에 적용한다.
