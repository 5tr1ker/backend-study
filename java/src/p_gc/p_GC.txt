[ 가비지 컬렉션이란? ]
GC는 JAVA 의 메모리 관리 방법중 하나로 Heap 영역에 동적으로 할당된 메모리 중 더이상 사용하지 않는 객체를 제거하는 프로세스이다.
C / C++ 언어와 다르게 개발자는 더이상 사용하지 않는 객체를 직접 해제하지 않아도 된다.

하지만 가비지 컬렉션은 자동으로 실행되기 때문에 언제 메모리가 언제 해제되는지 정확한 시간을 알 수 없으며,
가비지 컬렉션이 동작하는 동안에 다른 동작이 멈추는 Stop the World 가 발생한다.
만약 GC가 자주 발생하게 되면, 소프트웨어의 성능을 저하시킬 수 있다.

***
# - Stop-The-World
# JVM이 GC를 수행하기 위해 다른 작업을 멈추는 현상을 의미한다.
# JVM 은 GC를 수행하기 위해 GC 관련 쓰레드를 제외한 모든 쓰레드를 멈추게 되며, 이는 서비스 이용에 영향을 줄 수 있다.
***

따라서 어플리케이션의 사용성을 유지하면서, 효율적으로 GC를 수행하게 하는 GC 최적화 작업인 GC 튜닝이 중요하다.

[ 가비지 컬렉션 대상 ]
JVM은 Garbage 대상을 판단하기 위해서 도달성, 도달능력 ( Reachability ) 이라는 개념을 적용한다.
객체가 참조되고 있다면 Reachable , 객체가 참조되고 있지 않은 상태라면 Unreachable 로 구분한다. 이때 Unreachable 이 GC 대상이다.
# Unreachable : 접근 불가 상태 / Reachable : 접근 가능 상태

더 구체적으로, Heap Area 에 생성된 객체는 Method Area 이나, Stack Area 에서 해당 객체의 메모리 주소를 참조하게 된다.
하지만 메서드의 종료와 같은 이벤트로 Heap Area 에 있는 객체의 메모리 주소를 참조하는 참조 변수가 삭제되면, Unreachable 상태가 되는데
이때 이러한 객체들이 GC 대상이 되는 것이다.

[ 가비지 컬렉션 방식 Mark-And-Sweep ]
Mark-And-Sweep 는 사용하고 있는 객체를 솎아내는 내부 알고리즘이다.
즉, Root Space 로부터 해당 객체에 접근할 수 있는지 확인하며, 접근할 수 없으면 제거한다.
해당 방식은 가비지 컬렉션이 될 대상을 식별(Mark) 하고, 제거 (Sweep) 한 다음 파편화된 메모리 영역을 시작 지점으로 채워나가는 작업(Compact)을 수행한다.

Mark : Root Space 부터 시작하여 그래프 순회를 통해 연결된 객체 ( 참조하는 객체 ) 를 찾는다.
Sweep : Unreachable 객체를 제거한다.
Compact : Sweep 후 분산된 객체들을 Heap 의 시작주소로 모은다. ( 이는 GC 종류에 따라 하지 않을 수 있다. )
# 여기서 Root Space 는 Method Area , Stack Area , Native Method Stack , Static 변수 가 된다.
# Method Area 의 Static 변수 / Native Method Stack 의 JNI 참조

[ 가비지 컬렉션 동작 과정 ]
JVM 에서의 Heap Area에 저장되는 객체들은 대부분 일회성이며, 오랫동안 살아있는 객체는 드물다.
이러한 특성을 활용해 Heap 영역을 물리적으로 Young 영역과 Old 영역으로 나누었다.

- Young 영역
새롭게 생성되는 객체들이 할당되는 공간이며, 대부분의 객체들은 해당 영역에서 사라진다.
Young 영역에 대한 가비지 컬렉션은 Minor GC 라고 부른다.

- Old 영역
Young 영역에서 Reachable 상태를 유지한 객체가 복사되는 공간이다.
Young 영역보다 크기가 더 크게 할당되며, 가비지 컬렉션은 적게 발생된다.
Old 영역에 대한 가비지 컬렉션은 Major GC 또는 Full GC 라고 부른다.

# Old 영역이 Young 영역보다 큰 이유는, 보통 Young 영역은 생명 주기가 짧은 객체들은 큰 공간이 필요하지 않으며, 큰 객체는 바로 Old 영역으로 할당되기 때문이다.

또한 Young 영역은 3가지 영역으로 나뉜다. ( Eden , survivor 0 , survivor 1 )

- Eden
New 연산자를 통해 생성된 객체가 위치하며, 정기적인 가비지 컬렉션에서 살아남은 객체들은 survivor 영역으로 이동된다.

- Survivor 0 / Survivor 1
최소 1번 이상의 GC에서 살아남은 객체가 위치하며, Survivor 0 과 1 중 한 부분은 반드시 비워져 있어야 한다.

# 이렇게 힙 영역을 세부적으로 쪼갬으로써 가비지 컬렉터가 정확히 불필요한 객체만 제거하게 한다.
# 이때 각 분할된 부분은 객체의 생명주기를 면밀하게 관리한다.

[ Minor GC 과정 ]
Young Generation 은 객체가 처음 생성됐을 때 위치하는 메모리 공간이다.
Young Generation 은 Old Generation 보다 비교적 크기가 작기 때문에 메모리에서 객체를 찾아 제거하는 시간이 빠르다.

Minor GC 하는 과정은 다음과 같다.
1. new 연산자를 통해 생성된 객체는 Eden 영역에 배치된다.
2. Eden 영역이 차게되면 Minor GC 를 수행한다. ( Mark-And-Sweep )
3. GC 를 통해 살아남은 객체는 Survivor 영역으로 이동 ( 0 혹은 1 ) - 이때 살아남은 객체의 age 값이 1 상승한다.
4. 또다시 Eden 영역이 가득 차게 되면, Minor GC 를 수행한다. ( Mark-And-Sweep )
5. 그 후 Survivor 영역으로 이동하며 age 값을 1 상승시킨다. ( 이때 3번에서 Survivor 0 에 적재했다면, 객체들을 모두 Survivor 1 로 위치시킨다. , 즉 0과 1 번갈아가면서 적재 )

# Age 값이란?
# 객체가 Survivor 영역에서 살아남은 횟수를 의미하며, 임계치에 다다르면 Old 영역으로 편입될지 결정한다.
# JVM 에서 일반적인 HotSpot JVM은 임계치가 31이다. ( 6 bit )

또한 Survivor 영역은 2개의 영역 중에 1개의  영역은 반드시 비어있어야 한다. 만약 2개의 영역 모두 비어있거나 사용중이면 비정상적인 상태인 것이다.

[ Major GC 혹은 Full GC 과정 ]
Old Generation 영역은 Young Generation 영역에서 살아남은 객체들이 넘어오는 메모리 영역으로 age 값이 임계값을 되었을 경우에 이동된다.
Major GC는 Promotion 되어 ( Old 영역으로 넘어오는 현상 ) Old 영역의 메모리가 부족하게 되면 실행된다.
Major GC는 Minor GC보다 실행속도가 느리다.

# Promotion 는 Survivor 영역에서 객체의 age 값이 임계값이 도달하게 되면 Old 영역으로 넘어오는 현상이다.

Major GC의 과정은 다음과 같다.
1. Old Generation 영역에 할당된 메모리 허용치를 초과하게 되면 Major GC를 수행한다. ( Mark-And-Sweep )
2. 이때 Major GC는 모든 객체를 대상으로 참조되지 않은 객체를 제거한다.
# Major GC는 단순히 Old 영역이 가득차면 수행하는 단순한 방법이다.

Old 영역은 Young 영역보다 큰 공간을 가지고 있기 때문에 Major GC를 수행하는데 시간이 더 걸린다. ( 보통 Minor GC보다 10배 이상 )
이때 GC를 수행할 때 GC 관련 Thread 를 제외한 모든 쓰레드가 멈추는 Stop the World 현상으로 어플리케이션이 버벅이거나 느려질 수 있다.
또한 GC가 수행되면 CPU에 부하를 준다.

# Minor GC는 어플리케이션에 큰 영향을 주진 않는다.

[ 가비지 컬렉션 알고리즘 종류 ]

1. Serial GC
서버의 CPU 코어가 1개일 때 사용하는 GC으로, GC를 처리하는 쓰레드가 1개뿐이라 Stop the World 시간이 가장 길다.
Minor GC에는 Mark-Sweep 방식을, Major GC에는 Mark-Sweep-Compact 를 사용한다.
보통 실무에서 사용하는 경우는 없다.

설정 명령어 : java -XX:+UseSerialGC -jar Application.java

2. Parallel GC
Java 8 의 기본 GC이며, Miner GC는 멀티 쓰레드로 처리하고 Major GC는 싱글 쓰레드로 처리한다.
Serial GC처럼 Minor GC는 Mark-Sweep 방식을, Major GC에는 Mark-Sweep-Compact 방식을 사용한다.
기본적으로 GC를 처리하는 쓰레드는 CPU 코어 수와 동일하게 설정된다. ( 이는 설정 값을 통해 변경 가능 )

설정 명령어 : java -XX:+UseParallelGC -jar Application.java
-XX:ParallelGCThreads=N : 사용할 쓰레드의 갯수

3. Parallel Old GC
Parallel GC를 개선한 버전으로 Major GC도 멀티 쓰레드로 처리한다.
또한 Mark-Summary-Compact 방식으로 가비지 컬렉션을 수행한다.

설정 명령어 :  java -XX:+UseParallelOldGC -jar Application.java
# -XX:ParallelGCThreads=N : 사용할 쓰레드의 갯수

4. CMS GC ( concurrent Mark Sweep )
Stop the World 시간을 줄이기 위해 고안되었으며, 어플리케이션 쓰레드와 GC 쓰레드를 동시에 실행한다.
GC 과정이 복잡하기 때문에 다른 GC보다 CPU 사용률이 높다. 또한 메모리 파편화의 단점이 존재한다.
따라서 CMS GC Java9 부터 Deprecated 되었다가 Java14 부터는 사용이 중지되었다.

# CMS GC는 GC 대상을 찾기 위해 복잡한 여러 단계를 거쳐야 한다.

실행 명령어 : java -XX:+UseConcMarkSweepGC -jar Application.java

5. G1GC
CMS GC 를 대체하기 위해 나온 GC로, Java 9+ 부터 기본 GC로 사용되었다.
4GB 이상의 힙 메모리를 사용해야 하며, Stop the World 시간이 0.5초 필요할 때 사용한다. ( Heap Size가 작을 경우 미사용 권장 )

물리적으로 고정된 Old / Young 영역으로 나누어 사용하는 것이 아니라 Region 이라는 개념을 사용한다.
전체 힙 영역을 Region 이라는 영역으로 체스판처럼 나눈다음 각 영역의 역할을 상황에 따라 동적으로 할당하는 것이다. ( 역할은 Old , Eden , Survivor 등등.. )

G1GC는 수많은 Region 중에서 메모리가 가득찬 Region을 대상으로 GC를 수행한다.
즉, 전체 Heap 메모리를 대상으로 GC를 수행하는 것이 아닌, Region 대상으로 GC를 수행한다.
또한 기존 GC는 Eden->Survivor0->Survivor1 순서로 반복되지만 G1GC는 순차적으로 이동하지 않는다.
효율적이라고 판단하는 위치에 재할당(Reallocate) 한다. 즉 Survivor0 에 있는 객체가 Eden 으로 이동할 수 있다.

실행 명령어 : java -XX:+UseG1GC -jar Application.java

6. Shenandoah GC
Java 12 버전에 출시된 레드햇에서 만든 GC이다.
기존 CMS가 가지고 있는 단편화와 G1 가 가지고 있는 pause 문제를 해결했다.
효율적인 동시성과 가벼운 GC 로직을 통해 heap 사이즈에 상관없이 일정한 pause 시간이 소요된다.

실행 명령어 : java -XX:+UseShenandoahGC -jar Application.java

7. ZGC
Java 15 버전에 출시된 GC이다.
대량의 메모리 ( 8MB ~ 16TB ) 를 저-지연으로 처리하기 위해 만들어진 GC이다.
G1의 Region 처럼, ZGC는 ZPage 라는 영역을 사용하며, G1과는 다르게 각 영역이 동적인 크기이다. ( 2mb의 배수로 넓어짐 )
ZGC는 힙 사이즈가 커진다 하더라도 Stop the World 시간이 10ms 이상 소요되지 않는다.

실행 명령어 : java -XX:+UnlockExperimentalVMOptions -XX:+UseZGC -jar Application.java
