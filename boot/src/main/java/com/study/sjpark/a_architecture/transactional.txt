- 트랜잭션 ( @Transactional ) 동작 방식
트랜잭션을 의미하는 @Transactional 어노테이션은 AoP의 대표적인 활용 사례이다.
@Transactional 은 AoP 프록시를 통해 트랜잭션을 시작한 후, 커밋/롤백을 한다.

@Transactional 는 Advice Around 구조랑 동일하게 동작한다.
클라이언트 호출 -> 프록시 객체의 트랜잭션 시작 -> 프록시가 실제 객체 실행 -> 성공시 커밋 / 실패시 롤백

@Transactional 는 AoP 기반으로 구현되었기 때문에 프록시 객체가 없으면 동작하지 않는다.
또한 외부 호출만 AoP가 적용되기 때문에, 내부 호출은 AoP가 적용되지 않는다.

@Transactional 는 기본적으로 RuntimeException 만 롤백된다. 필요시 아래 코드처럼 Check Exception 도 롤백 범위로 설정할 수 있다.
@Transactional(rollbackFor = Exception.class)

그리고 트랜잭션은 DB 연결 단위이기 때문에 메서드 호출이 종료되어야 커밋되어 DB에 반영이 된다.

- 내부 호출은 AoP가 적용되지 않는 이유 ( self-invocation )
AoP를 활용할 때 외부 호출만 AoP가 적용이 되는데, 여기서 범위는 AoP가 적용된 Bean 객체이다.
만약 실제 객체 안에서 내부 호출을 하면, 이미 프록시를 거친 상태이기 때문에 다시 프록시를 거치지 않는다.
만약 다른 Bean 에서 AoP가 적용된 Bean을 호출하면 프록시를 거치게 된다.

즉, 프록시 객체는 외부에서 들어온 요청만 처리된다. 내부 객체는 이미 프록시 안에서 처리되고 있기 때문이다.

@Service
public class UserService {

    public void outer() {
        inner(); // 트랜잭션 inner 호출
    }

    @Transactional
    public void inner() {
        // 트랜잭션 적용 안됨
    }
}
위의 예제는 inner() 메서드에서 트랜잭션이 적용되지 않는다. 그 이유는 outer() 메서드를 호출하고 있는 시점은 이미 프록시 객체를 지나 실제 객체로 들어와있기 때문이다.
이는 실제 객체 안에서의 호출이기 때문에 프록시의 영향을 받지 않는 것이다.

물론 outer() 를 호출할 때에도 프록시 객체를 통해 실제 객체로 들어온다. 다만 outer() 는 @Transactional 이 적용이 안되어 Advice 가 적용되지 않고 실제 객체로 들어온다.

@Transactional
public void outer() {
    inner(); // 트랜잭션 inner 호출
}
@Transactional
public void inner() {
    // 트랜잭션 적용 안됨
}
예제를 위와 같이 수정할 경우에 outer() 호출 시 트랜잭션이 실행되고, inner() 까지 처리한 뒤에 커밋된다.
즉, 트랜잭션이 outer() 와 inner() 개별적으로 실행되는 것이 아니라 하나의 트랜잭션을 공유하게 된다.

정리하면 @Transactional 가 적용 되는 것은 어노테이션을 메서드에 명시한 것이 아니라 "프록시를 통과해야 적용된다."
프록시를 통과해야 한다는 것은 Proxy 객체를 타고 들어오는 것을 말하며, 내부 호출은 Proxy 객체를 타고 들어오는 것이 아닌 타겟 객체를 직접 호출하는 것이기 때문에
AoP가 적용되지 않는다. ( 구조는 클라이언트 -> Proxy 객체 -> 타겟 객체 로 되어있다. )

# 이러한 경우 자기 자신을 주입하거나 구조 분리하는 것이 좋다. 즉, 항상 외부 호출로 만들면 된다.

- 트랜잭션 전파 ( Propagation )
트랜잭션 전파는 이미 트랜잭션이 실행중일 때 새로운 트랜잭션을 어떻게 처리할지 정의하며, 반드시 프록시를 통해 호출되어야 적용된다.

1. @Transactional ( 있으면 합류, 없으면 생성 - 기본값 )
가장 많이 쓰는 옵션이다.

2. @Transactional(propagation = REQUIRES_NEW) ( 무조건 새 트랜잭션 생성 )
기존 트랜잭션이 실행 중이면 일시 중지하고, 새 트랜잭션을 생성한다.
이후 새로운 트랜잭션이 종료되면 기존 트랜잭션을 다시 실행한다.
이는 완전히 독립된 트랜잭션이다.

# 만약 내부 호출인 경우에 기존 트랜잭션을 활용한다. 그 이유는 프록시가 적용되지 않기 떄문에 트랜잭션 전파도 적용되지 않는다.

3. @Transactional(propagation = SUPPORTS)
트랜잭션이 있으면 참여하고, 없으면 트랜잭션 없이 실행한다.

4. @Transactional(propagation = MANDATORY)
트랜잭션이 "없으면" 예외를 발생시킨다.

5. @Transactional(propagation = NEVER)
트랜잭션이 "있으면" 예외를 발생시킨다.

6. @Transactional(propagation = NOT_SUPPORTED )
트랜잭션을 비활성화 시키고 실행한다.
