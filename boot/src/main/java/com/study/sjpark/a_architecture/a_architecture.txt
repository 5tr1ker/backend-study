[ Spring Boot 와 Spring Framework 의 차이 ]
Spring Framework는 JAVA 기반의 웹 어플리케이션 프레임워크이며,
Spring Boot는 Spring Framework 위에서 동작하며, 자동 설정과 실행 환경을 제공하는 확장 프레임워크이다.

- 서버 생성 및 구동 시 설정
Spring Framework는 서버를 구동하기 위한 설정 값 ( XML , Config , DispatcherServlet, ViewResolver 등을 모두 등록하며,
WAR로 패키징을 통해 외부 WAS서버 ( ex 톰캣 ) 에 배포하여 서버를 구동하는 방식이다.
혹은 설정을 통해 Spring Framework 도 Embedded Server ( 톰캣 ) 에서 구동할 수 있다.

그에 반해 Spring Boot는 이러한 설정을 자동화 해주며, 내장 서버를 가지고 있어 프로젝트 생성과 동시에 서버를 구동시킬 수 있다.
물론 설정 값들은 개발자가 설정할 수 있으며, 설정하지 않으면 기본 값을 가지고 구동된다.

- 의존성 관리
Spring Framework 는 라이브러리를 사용할 때, 버전을 맞춰주어야 한다.
즉, 라이브러리를 추가하기 위해 다른 라이브러리와 버전이 충돌하는지 직접 확인하며 추가해야 한다.
이는 최악의 경우 버전 충돌을 위해서 많은 시간을 할애해야 한다.

Spring Boot는 의존성에 대해 버전을 명시하지 않아도, 자동으로 버전을 맞춰준다.
버전 명시를 하면 해당 버전으로 설치가 진행되지만 버전을 명시하지 않으면 다른 라이브러리와 호환되는 버전으로 설정해준다.

- 운영 관리 편의
Spring Boot에는 헬스 체크, 메트릭, 외부 설정파일 ( application.yml ) , 프로파일 관리를 통해 개발 뿐만이 아니라 운영에서도
편리하게 운영할 수 있도록 여러 기능을 제공한다.

1. 헬스 체크 ( Spring Boot Actuator )
Spring Boot에서는 Spring Boot Actuator 라이브러리를 제공해서 현재 시스템 뿐만 아니라 디스크, 외부 연동 시스템 , DB 연동 ,
Kafka Queue 까지 Health Check 할 수 있다.
의존성을 추가해주면 큰 설정 없이 활용할 수 있다.
Spring Boot와 의존성이 강하기 때문에 Spring Framework 에서 활용하기 위해 endpoint 등록 및 설정 변경 등
많은 설정 작업이 필요하다.

/actuator/health 를 호출하여 다양한 정보를 확인할 수 있다.
헬스 체크는 단순히 해당 시스템이 살아있냐 / 정상 동작하는지를 알 수 있다.

2. 메트릭 ( Spring Boot Actuator )
메트릭이란 시스템 상태를 수치로 표현한 데이터를 말한다.
Spring Boot Actuator 라이브러리는 헬스 체크 뿐만이 아니라 시스템 상태를 수집 및 수치화 해서 보여준다.

대표적으로 HTTP 요청 ( 요청 수 , 응답 시간 , 상태 코드 별 통계 ) , JVM ( GC 상태 , Thread 상태 , Heap 메모리 상태 ) ,
시스팀 ( CPU와 디스크 사용량 ) , DB와 커넥션 풀 ( 커넥션 사용량 , 대기 수 ) 등에 대한 다양한 정보를 API 하나로 쉽게 확인할 수 있다.

/actuator/metrics 를 호출하여 다양한 정보를 확인할 수 있다.
메트릭은 시스템이 얼마나 잘 돌아가고 있는지에 대한 정보를 쉽게 알 수 있다.

활용 예제로는 메트릭과 Grafana , Prometheus 를 연결하여 외부 모니터링 시스템과 연동해서 모니터링한다.

# 메트릭 또한 Spring Framework 에서도 구현할 수 있지만, 직접 수집하고 반환하는 API를 개발해야 한다.
# 즉 Spring Actuator 는 Spring Boot 환경에 친화적이여서 Spring Framework 와 연동하기 위해선 많은 설정 작업이 필요하다.

3. 외부 설정 파일 ( application.yml ) 과 프로파일 관리
Spring Framework 에서도 application.yml 을 사용할 수 있으나, 내부 값을 코드로 활용하려면 직접 값을 가져와야 한다.
간단히 말하면 Spring Framework 는 yml 를 기본 포맷으로 지원하지 않아, 직접 파싱하고 Bean을 직접 등록해주어야 한다.

Spring Boot는 yml을 표준으로 지원하고 있어 @Value, @ConfigurationProperties 어노테이션을 활용하면 간단하게
값을 가져올 수 있으며, DataSource 나 kafka를 의존성에 추가했을 때 yml에 연결 정보를 입력하면 간단하게 연동할 수 있다.

Spring Framework 였다면 연결 정보를 등록하기 위해 yml 과 연동하기 위한 Bean 을 등록하고, 직접 파싱하고 DataSource를
제공하는 별도의 클래스를 구현해야 했을 것이다.

# 다만 Spring Framework 는 .properties 나 XML 은 기본으로 지원한다.

또 한가지 장점으로는 Spring Boot는 프로파일을 제공하여 프로필에 따라 각기 다른 설정을 적용할 수 있다.
application.yml
application-local.yml
application-dev.yml
application-prod.yml
위와 같이 다양한 환경에 대비해 여러 설정파일들을 미리 적용해두어, 기동 시점에 어떤 설정 파일을 적용할지 적용할 수 있다.
--spring.profiles.active=dev 옵션을 활용하면 -dev 가 붙은 .yml 이 적용되어 서버가 구동이 된다.

매우 중요한 사실은 프로파일이 지정되지 않은 경우 application.yml 만 적용되지만, 프로파일이 dev 인 경우 application.yml
와 application-dev.yml 가 적용된다. 즉, 2개의 yml 설정 파일이 적용되며 application.yml 와 application-dev.yml 가
중복되는 키 값이 있으면 application-dev.yml 로 오버라이딩 된다.
즉, application.yml 는 기본으로 무조건 적용된다.

# application.yml 내부에 spring.profiles.active=dev 으로 설정할 수 있지만, 이렇게 되면 빌드된 후에는 변경할 수 없어 잘 사용하지 않는다.

그리고 밑의 옵션 값과 같이 여러개의 옵션을 적용할 수 있으며, 코드 단에서 프로필에 따라서 빈을 서로 다른 것을 등록할 수 있다.
--spring.profiles.active=dev,db
< 프로필에 따른 빈 등록 >
@Profile("dev")
@Bean
public DataSource devDataSource() { ... }

- Auto configuration
Spring Boot는 내부적으로 @Conditional 나 @EnableAutoConfiguration 가 등록되어 있다면,
classpath 나 Bean 등록 여부를 확인하여 조건에 맞으면 빈을 등록해준다.
