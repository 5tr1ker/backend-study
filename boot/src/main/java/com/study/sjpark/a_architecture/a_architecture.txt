[ Spring Boot 와 Spring Framework 의 차이 ]
Spring Framework는 JAVA 기반의 웹 어플리케이션 프레임워크이며,
Spring Boot는 Spring Framework 위에서 동작하며, 자동 설정과 실행 환경을 제공하는 확장 프레임워크이다.

- 서버 생성 및 구동 시 설정
Spring Framework는 서버를 구동하기 위한 설정 값 ( XML , Config , DispatcherServlet, ViewResolver 등을 모두 등록하며,
WAR로 패키징을 통해 외부 WAS서버 ( ex 톰캣 ) 에 배포하여 서버를 구동하는 방식이다.
혹은 설정을 통해 Spring Framework 도 Embedded Server ( 톰캣 ) 에서 구동할 수 있다.

그에 반해 Spring Boot는 이러한 설정을 자동화 해주며, 내장 서버를 가지고 있어 프로젝트 생성과 동시에 서버를 구동시킬 수 있다.
물론 설정 값들은 개발자가 설정할 수 있으며, 설정하지 않으면 기본 값을 가지고 구동된다.

- 의존성 관리
Spring Framework 는 라이브러리를 사용할 때, 버전을 맞춰주어야 한다.
즉, 라이브러리를 추가하기 위해 다른 라이브러리와 버전이 충돌하는지 직접 확인하며 추가해야 한다.
이는 최악의 경우 버전 충돌을 위해서 많은 시간을 할애해야 한다.

Spring Boot는 의존성에 대해 버전을 명시하지 않아도, 자동으로 버전을 맞춰준다.
버전 명시를 하면 해당 버전으로 설치가 진행되지만 버전을 명시하지 않으면 다른 라이브러리와 호환되는 버전으로 설정해준다.

- 운영 관리 편의
Spring Boot에는 헬스 체크, 메트릭, 외부 설정파일 ( application.yml ) , 프로파일 관리를 통해 개발 뿐만이 아니라 운영에서도
편리하게 운영할 수 있도록 여러 기능을 제공한다.

1. 헬스 체크 ( Spring Boot Actuator )
Spring Boot에서는 Spring Boot Actuator 라이브러리를 제공해서 현재 시스템 뿐만 아니라 디스크, 외부 연동 시스템 , DB 연동 ,
Kafka Queue 까지 Health Check 할 수 있다.
의존성을 추가해주면 큰 설정 없이 활용할 수 있다.
Spring Boot와 의존성이 강하기 때문에 Spring Framework 에서 활용하기 위해 endpoint 등록 및 설정 변경 등
많은 설정 작업이 필요하다.

/actuator/health 를 호출하여 다양한 정보를 확인할 수 있다.
헬스 체크는 단순히 해당 시스템이 살아있냐 / 정상 동작하는지를 알 수 있다.

2. 메트릭 ( Spring Boot Actuator )
메트릭이란 시스템 상태를 수치로 표현한 데이터를 말한다.
Spring Boot Actuator 라이브러리는 헬스 체크 뿐만이 아니라 시스템 상태를 수집 및 수치화 해서 보여준다.

대표적으로 HTTP 요청 ( 요청 수 , 응답 시간 , 상태 코드 별 통계 ) , JVM ( GC 상태 , Thread 상태 , Heap 메모리 상태 ) ,
시스팀 ( CPU와 디스크 사용량 ) , DB와 커넥션 풀 ( 커넥션 사용량 , 대기 수 ) 등에 대한 다양한 정보를 API 하나로 쉽게 확인할 수 있다.

/actuator/metrics 를 호출하여 다양한 정보를 확인할 수 있다.
메트릭은 시스템이 얼마나 잘 돌아가고 있는지에 대한 정보를 쉽게 알 수 있다.

활용 예제로는 메트릭과 Grafana , Prometheus 를 연결하여 외부 모니터링 시스템과 연동해서 모니터링한다.

# 메트릭 또한 Spring Framework 에서도 구현할 수 있지만, 직접 수집하고 반환하는 API를 개발해야 한다.
# 즉 Spring Actuator 는 Spring Boot 환경에 친화적이여서 Spring Framework 와 연동하기 위해선 많은 설정 작업이 필요하다.

3. 외부 설정 파일 ( application.yml ) 과 프로파일 관리
Spring Framework 에서도 application.yml 을 사용할 수 있으나, 내부 값을 코드로 활용하려면 직접 값을 가져와야 한다.
간단히 말하면 Spring Framework 는 yml 를 기본 포맷으로 지원하지 않아, 직접 파싱하고 Bean을 직접 등록해주어야 한다.

Spring Boot는 yml을 표준으로 지원하고 있어 @Value, @ConfigurationProperties 어노테이션을 활용하면 간단하게
값을 가져올 수 있으며, DataSource 나 kafka를 의존성에 추가했을 때 yml에 연결 정보를 입력하면 간단하게 연동할 수 있다.

Spring Framework 였다면 연결 정보를 등록하기 위해 yml 과 연동하기 위한 Bean 을 등록하고, 직접 파싱하고 DataSource를
제공하는 별도의 클래스를 구현해야 했을 것이다.

# 다만 Spring Framework 는 .properties 나 XML 은 기본으로 지원한다.

또 한가지 장점으로는 Spring Boot는 프로파일을 제공하여 프로필에 따라 각기 다른 설정을 적용할 수 있다.
application.yml
application-local.yml
application-dev.yml
application-prod.yml
위와 같이 다양한 환경에 대비해 여러 설정파일들을 미리 적용해두어, 기동 시점에 어떤 설정 파일을 적용할지 적용할 수 있다.
--spring.profiles.active=dev 옵션을 활용하면 -dev 가 붙은 .yml 이 적용되어 서버가 구동이 된다.

매우 중요한 사실은 프로파일이 지정되지 않은 경우 application.yml 만 적용되지만, 프로파일이 dev 인 경우 application.yml
와 application-dev.yml 가 적용된다. 즉, 2개의 yml 설정 파일이 적용되며 application.yml 와 application-dev.yml 가
중복되는 키 값이 있으면 application-dev.yml 로 오버라이딩 된다.
즉, application.yml 는 기본으로 무조건 적용된다.

# application.yml 내부에 spring.profiles.active=dev 으로 설정할 수 있지만, 이렇게 되면 빌드된 후에는 변경할 수 없어 잘 사용하지 않는다.

그리고 밑의 옵션 값과 같이 여러개의 옵션을 적용할 수 있으며, 코드 단에서 프로필에 따라서 빈을 서로 다른 것을 등록할 수 있다.
--spring.profiles.active=dev,db
< 프로필에 따른 빈 등록 >
@Profile("dev")
@Bean
public DataSource devDataSource() { ... }

- Auto configuration
Auto Configuration은 어플리케이션의 실행 환경 구성과 설정을 자동으로 해주는 기능이다.
classPath에 존재하는 라이브러리와 설정 파일 ( yml ) , Bean 등록 여부를 기준으로 조건에 맞는 Bean을 자동으로 등록하는 기능이다.
# classPath는 JVM이 클래스나 리소스를 찾는 경로의 집합이다.

이를 통해 DataSource, JPA , MVC 설정 등 어플리케이션 구동 시 필요한 설정을 자동으로 구성해주며,
서버 구동 시 Embedded Tomcat과 DispatcherServlet 또한 자동으로 설정된다.

물론 개발자가 Bean 을 직접 등록하여 커스터마이징을 할 수 있도록 지원을 한다.
즉, Spring Boot는 기본적인 설정을 자동으로 해주며, 개발자가 필요한 부분만 커스터마이징할 수 있다.

[ Spring architecture / 요청 처리 흐름 ( Request Lifecycle ) ]

Spring 은 크게 Servlet Container 영역과 Spring Container 영역으로 나뉜다.

1. Servlet Filter
Servlet은 모든 요청을 필터링하며 Spring 과 무관한 Servlet 영역이다.
사용자의 요청을 가장 먼저 처리하며, 보통 인증/인코딩 설정/로깅을 담당한다.

2. DispatcherServlet
Spring MVC의 시작 입구이며, 모든 요청을 중앙에서 제어한다.
요청을 어떤 Controller 에게 위임할 지 결정을 한다.

3. HandlerMapping
요청으로 넘어온 URL 을 어떤 컨트롤러에서 처리될 수 있는지 정보를 반환한다.
구체적으로 요청된 URL 에 해당되는 Controller 객체를 찾아 DispatcherServlet 에게 반환한다.

ex ) @GetMapping("/users") 을 기준으로 탐색한다.

4. HandlerAdapter
Controller 를 실행할 수 있게 연결해줄 수 있는 어댑터이다.
이때 다양한 형태의 Controller 를 호출할 수 있게 유연성을 가지고 있다.

왜냐하면 Controller 는 Annotation이나 HttpRequestHandler 등 다양한 형태를 가지고 있기 때문이다.

5. Interceptor ( PreHandler )
Controller 가 실행되기 전에 호출된다.
비즈니스 로직과 분리된 공통 로직을 처리한다. 예를 들면 권한 체크, 로그인 검사, 로깅 등을 수행할 수 있다.

6. Controller
개발자가 작성한 비즈니스 로직이 수행되는 공간이다. Service 나 Repository가 실행되어 DB에 적재하거나,
사용자 요청에 맞는 데이터를 조회해서 결과를 반환한다.

7. Interceptor ( postHandler )
컨트롤러가 실행된 직후, 뷰 ( View ) 를 렌더링하기 전에 호출된다.
컨트롤러가 반환한 데이터를 가공하거나 공통 작업을 처리할 수 있다.

8. View Resolver / Message Converter
View Resolver : 사용자에게 화면을 보여주어야 할 때, 어떤 JSP나 Thymeleaf 를 보여줄지 결정한다.
Message Converter ( @RestController ) : 사용자에게 데이터를 직접 응답해야 할 때, JSON 이나 XML 으로 변환한다.

9. Interceptor ( afterCompletion )
모든 처리가 끝난 후에 호출된다.
요청 처리 과정에서 사용된 리소스나 실행 시간 등을 기록하기 위해 활용된다.

10. Response
사용자에게 최종 결과를 반환한다.
JSON 이나 HTML 등

[ WAS 와 WS 차이 ]
WS는 Web Server 의 약자로 HTTP 요청을 받아 정적 리소스 처리 및 앞단 역할을 수행한다.
대표적으로 NginX 와 Apache HTTP Server 가 있다.
주로 HTML,CSS,JS,이미지와 같은 정적 파일을 응답하거나 Reverse Proxy 및 Load Balancing 역할을 한다.

- 보안 / 로드 밸런싱 / Reverse Proxy
WS 는 주로 WAS 앞단에서 부하 분산 및 보안 강화를 위해 활용된다.
보안 : TLS 종료 처리 , 인증서 관리 , 기본적인 요청 필터링 ( WS에서 HTTPS 처리 후 내부에서는 HTTP 처리 )
Reverse Proxy : 클라이언트 대신 WAS 에게 요청 전달 , 내부 서버 구조 은닉 , 공격 / 비정상적인 요청 차단 ( 사용자는 WAS로 요청을 하는 것이 아닌 WS 로 요청을 한다. )
로드 밸런싱 : 여러 WAS 대상으로 트래픽 분산 , 장애 서버 제외 ( 헬스 체크 )
정적 리소스 처리 : 이미지,CSS,JS는 WAS를 안거치고 직접 응답하여 성능 향상

WAS 는 Web Application Server 의 약자로 동적 처리와 어플리케이션 실행을 담당한다.
대표적으로 Apache Tomcat , Jetty , Undertow 가 있으며, 사용자 요청에 따른 동적 응답을 제공하는데 활용된다.
Servlet 기반으로 동작하며, 비즈니스 로직 수행 및 DB 연동을 통해 동적 응답을 생성한다.

[ Servlet Container ]
Servlet Container 는 Servlet 을 실행하기 위한 환경이며, Servlet의 생명주기 관리와 요청 처리를 담당한다.
이러한 Servlet Container 기능을 포함한 WAS는 대표적으로 Apache Tomcat가 있으며, 그 외에 Jetty와 Undertow 도 있다.
Spring Boot는 기본적으로 Apache Tomcat을 내장 서버로 사용한다.

Apache Tomcat는 클라이언트의 HTTP 요청을 수신하고 이를 처리할 Servlet을 실행하며 응답을 반환한다.

Tomcat은 웹 서버와 Servlet Container를 포함하고 있는 WAS 서버로
Servlet Container는 어플리케이션 실행 시 or 최초 요청 시 Servlet 객체를 생성하고, 요청이 들어오면 Servlet 객체를 재활용하여 요청을 처리한다.
# 즉, Servlet 객체는 싱글톤이며 여러 쓰레드가 공유하여 처리한다.
# 여기서 말하는 Servlet 의 구현체는 DispatcherServlet 이다.

Servlet Container 를 포함하고 있는 Tomcat ( WAS ) 는 아래와 같은 역할을 담당한다.
1. 웹 서버와 통신 : Tomcat은 Connector를 통해 HTTP 요청 및 소켓 연결을 처리하고, 수신받은 데이터인 네트워크 데이터를 자바 객체인
HttpServletRequest 와 HttpServletResponse 으로 변환하여 Servlet Container 에 전달한다.
2. 필터 : DispatcherServlet 에 진입하기 전/후 Servlet Container 영역에서 요청에 대한 부가 작업을 처리한다.
3. 웹 어플리케이션 실행 : JSP나 Spring 기반 어플리케이션을 실행한다.
4. 쓰레드 관리 : 한 요청당 하나의 쓰레드가 할당되며 병렬로 처리된다 ( 멀티 쓰레드 ) . 그리고 Tomcat은 이러한 쓰레드들은 Thread Pool 에서 관리된다.
5. 요청/응답 객체 관리 : 요청 데이터를 HttpServletRequest 와 HttpServletResponse 객체로 생성하고 관리한다.
6. 세션/보안 처리 : HTTP 세션을 생성 및 관리하며, 쿠키/세션 기반 인증에 대한 기본적인 보안 기능을 제공한다.

그 외 Servlet Container는 DispatcherServlet 을 생성하고 생명주기를 관리한다. ( DispatcherServlet 등록은 Spring Boot 가 한다. )

즉, 톰캣은 네트워크 처리부터 Servlet 실행, 쓰레드와 세션 관리까지 어플리케이션 실행에 필요한 전반적인 것들을 관리한다.

- 서블릿 ( Servlet )
JAVA Servlet 스펙에 따라 만들어진 서버 측 컴포넌트이다. HTTP 요청을 처리하는 자바 기반의 기술이다.
이 Servlet 인터페이스를 구현한 클래스가 DispatcherServlet 이다.

이러한 DispatcherServlet 은 웹 계층의 진입점이며, 요청 처리 흐름을 제어하는 역할을 한다.

- JAVA Servlet 스펙
JAVA Servlet 스펙은 웹 요청 처리를 위한 자바 표준 규칙을 말한다.
Servlet 인터페이스는 init , service , destroy 의 생명 주기와 HTTP 요청 데이터를 HttpServletRequest / HttpServletResponse 객체로 처리하는 방법을 정의한다.

이러한 스펙이 중요한 이유는 구현체를 바꿔도 코드가 그대로 동작하게 하기 위해서이다.
ex ) Apache Tomcat , Jetty , Undertow

- Servlet의 생명 주기
Servlet Container 는 Servlet의 생명주기를 관리하는데, 여기서 생명주기는 총 3가지가 있다.
init ( 초기화 ) : Servlet Container가 Servlet 객체를 생성하고 단 한번만 호출한다.
service ( 요청 처리 ) : HTTP 요청이 들어올 때 마다 호출되며, 내부 비즈니스 로직을 수행한다. 이때 멀티쓰레드 환경으로 실행된다.
destroy ( 종료 단계 ) : 서버 종료 혹은 Servlet 객체 제거 시 단 한 번 호출된다.

여기서 init 과 destroy 단계에서는 Servlet Container 는 호출만 할 뿐 사실은 내부가 비어있다.
그래서 개발자는 아래와 같이 Servlet의 구현체인 HttpServlet를 상속받아 init() 과 destroy() 를 구현할 수 있다.
물론 이는 선택 사항이며, 필요시에 구현하면 된다.

public class MyServlet extends HttpServlet {

    @Override
    public void init() {
        // 기본적으로 비어 있음
    }

    @Override
    public void destroy() {
        // 기본적으로 비어 있음
    }
}

# 이때 Servlet은 단 하나만 생성이되며 모든 쓰레드가 이 Servlet 객체를 공유하여 사용한다.
# 그렇기 때문에 Thread-Safe 하지 않으면 장애가 발생한다.

[ Spring Container ]
Spring Container 영역은 스프링 빈을 생성하고 생명주기를 관리하며, 객체 간의 의존성 주입을 담당한다.
그리고 아래와 같은 특징이 있다.
1. 객체 관리 : Controller, Service, Repository 등 어플리케이션에서 사용하고 있는 객체(Bean)들를 생성하고 관리한다.
2. 의존성 주입 : 객체 간의 의존 관계를 자동으로 연결하여 결합도를 낮춘다.
3. 생명주기 관리 : Bean의 생성, 초기화, 소멸까지 전체 생명주기를 관리한다.
4. 유연성 : POJO ( Plain Old Java Object ) 기반으로 특정 기술에 종속되지 않는 객체를 사용할 수 있다.
5. 인터셉터 : Controller에게 요청이 전달되기 전과 후에 부가 작업을 처리한다.

[ Filter vs Interceptor ]
Filter는 Servlet Container 영역에 위치하여, DispatcherServlet 에게 요청이 전달되기 전에 가장 먼저 처리한다.

Interceptor는 Spring Container 내부에 위치하고 있으며 Controller가 호출되기 전과 후에 호출된다.
인증과 권한, 로깅에 활용되며 컨트롤러에게 요청이 전달되기 전과 후에 요청과 응답에 대해 부가 작업을 처리한다.
