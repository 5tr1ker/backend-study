[ Kafka 의 구성요소 ]
Kafka는 분산 로그 기반 메세징 시스템으로, 고성능이며 확장성이 좋다는 특징을 가지고 있다.
Kafka의 전체 구조로는 Producer -> Topic -> Partition -> Broker -> Consumer 로 구성되어 있다.

1. Producer ( 생산자 )
데이터를 Kafka로 보내는 주체로 API 서버나 로그 수집기가 될 수 있다.
Producer는 메세지를 특정 Topic 으로 전달하고, 어떤 Partition에 넣을지를 결정한다.
이때 Partition에 넣을 때는 특정 Key를 기반으로 넣을 수 있고, Round Robin 으로 넣을 수 있다.

2. Topic ( 논리적 채널 )
메세지를 논리적으로 분류하는 단위이며 실제 저장은 Topic 안에 Partition 에 저장된다.
그렇기 때문에 Topic은 최소 1개 이상의 Partition을 가져아 한다.

3. Partition ( 파티션 )
Partition은 독립적인 로그 파일이며 Topic을 쪼개놓은 단위이다.
1개의 Partition은 하나의 Topic에 매핑되며, 분류된 데이터가 실제로 저장이 된다.
# 반대로 1개의 Topic은 여러개의 Partition을 가질 수 있다.

그리고 같은 Partition 에 저장된 메세지는 순서가 보장이 되며 Consumer는 Offset을 기반으로 메세지를 읽는다. ( index 와 유사 )
한 Topic에 여러 Partition을 구성하여 분산 저장을 통해 병렬 처리를 할 수 있다.

이렇게 한 Topic에 대해 여러 Partition 이 구성되었을 때, 메세지는 해당 Partition 중에서 하나만 들어가게 된다.
이때 특정 Partition 에 넣기 위한 기준으로는 Round-Robin , Key 기반 방식이 있다.
# 한 메세지를 조각내어 여러 Partition에게 저장하는 것은 아니다.

- Round Robin ( 기본 값 )
Key를 사용하지 않은 경우 기본 값이 되며 각각의 Partition 에게 균등 분배된다.
그렇기 때문에 전체 작업을 기준으로 순서가 보장되지 않는다.

- Key 기반 분류
같은 Key는 같은 Partition에 할당된다.
이를 활용하여 특정 기준으로 순서를 보장할 수 있으며, 해당 방법을 가장 많이 활용한다.

위와 같이 여러 Partition을 배치하면 병렬 처리를 통해 성능이 향상되고 확장성이 좋지만, 순서가 보장되지 않을 수 있다는 단점이 존재한다.
다만 같은 Key를 활용하면 어느정도 순서는 보장할 수 있기 때문에 시스템 구조에 따라 유연하게 결정하면 된다.
# Partition의 갯수는 Consumer의 갯수, TPS , 순서 보장 필요 여부에 따라 결정하면 된다.
# 또한 순서가 중요할 경우 Partition을 1개만 쓰거나 같은 Key를 활용하고, 성능이 중요한 경우 여러 Partition 을 배치하여 병렬 처리로 배치하면 된다.

그리고 Key의 갯수가 Partition의 갯수보다 많을 경우 1개의 Partition에 다수의 Key가 매핑될 수 있다.
Kafka는 Partition을 아래와 같이 해시 기반 분배한다.
Partition = hash(key) % partition_count

해시 기반 분배이기 때문에 각 Partition에 대해 균등 분배는 보장되지 않는다.
즉, Key의 갯수가 Partition 보다 크거나 같더라도 Key의 해시 값에 의해 비어있는 Partition이 존재할 수 있다.
K1 = P0
K2 = P2
K3 = P2
K4 = P0
( 비어있는 P1 ... )
# Hash의 결과 값은 예상할 수 없다.

Key의 갯수가 적을 경우 특정한 Partition에 데이터가 몰릴 수 있지만 key의 수가 많아질수록 해시 분포가 균등해진다.
그렇기 때문에 Key의 수가 적으면 partition imbalance 가 발생하고, consumer 병렬성이 깨질 수 있다.

- Partition의 저장 방식
1개의 Partition 당 여러 개의 로그 파일로 나누어 저장한다.
즉, 한개의 큰 파일이 아닌 작은 파일 단위인 세그먼트로 나누어 관리한다.
파일 저장 시 .log ( 실제 메세지 데이터 ) , .index ( offset 위치 매핑 ) , .timeindex ( timestamp 기반 조회 ) 로 나누어 저장한다.
또한 위의 Segment는 일정 시간이 지나거나 일정 크기가 넘어서면 새 파일에 저장한다. 그래서 1개의 Partition 에는 N * 3 개의 Segment 파일이 생성된다.

이렇게 파일을 쪼개는 데에는 성능과 크게 연관이 있다.
쓰기 최적화 : Kafka는 append-only 로 파일의 뒤에서만 쓰는 Sequential I/O 방식이기 때문에 매우 빠르다.
관리 최적화 : Kafka는 TTL / retention 기반으로 오래된 Segment를 통째로 삭제한다.
장애 최소화 : 파일이 깨질 경우 전체 장애 영향을 최소화한다.

4. Broker ( 서버 )
Kafka 서버 1대당 Broker 서버 1대가 할당이되며, 메세지를 저장하고 Producer/Consumer 요청을 처리하며 Partition 분산, 복제 관리와 장애 대응까지
담당한다. Broker는 실제 물리적 리소스이기 때문에 Partition 을 나눠 병렬 처리를 한다 하더라도 물리적 리소스가 따라주지 못하면 성능은
크게 향상되지 못한다.

브로커가 하는 역할을 구체적으로 정리하면 다음과 같다.

- 메세지 저장
Partition 데이터를 디스크에 로그 형태로 저장한다.
이때 append-only 구조로 파일 뒤에서만 쓰는 Sequential I/O 방식을 활용하여 빠르게 쓴다.

- Producer 요청 처리
Producer가 메세지를 보내면, 특정 Partition에 메세지를 저장한다.

- Consumer 요청 처리
Consumer가 메세지를 읽을 때, Offset을 기준으로 메세지를 전달한다.

- Partition 관리
여러 브로커가 구성이 되었을 경우, Partition을 여러 브로커가 나눠서 독립적으로 관리한다.
그 이유는 부하를 분산하기 위함이며, Partition 단위로 병렬처리하기 위함이다.
또한 Leader 와 Follower 를 나눠 배치함으로 써 장애가 발생해도 대응할 수 있다.

- Leader/Follower 관리 ( 복제 관리 ) 및 Partition 리더 선출 ( 장애 대응 )
하나의 Partition에는 1개의 Leader 와 N개의 Follower 가 존재한다.
Leader는 실제로 읽기/쓰기를 담당하고 Follower 는 Leader 의 상태를 복제하기만 한다.
# Producer/Consumer는 평소에 Leader와만 소통을 한다.
이때 Leader가 죽게 되면 Follower 중에서 1개를 Leader 로 승격시킨다.

만약 다중 Broker로 구성이 되어있다면, 여러 Broker에 의해 분산되어 관리된다.
ex )
Broker 0 - Leader 1 / Follower 2
Broker 1 - Leader 2 / Follower 0
Broker 2 - Leader 0 / Follower 1

- 복제 관리 (Replication Management)
ISR (In-Sync Replicas) : Leader와 동일한 상태인 Follower들의 집합이다.
데이터 복제 방식 : Leader 에 메세지가 저장되면 Follower는 Pull 방식으로 메세지를 복제한다.
ack 설정 : acks=0 ( Producer가 메세지를 보내고 종료 ) , acks=1 ( Leader만 저장되면 OK ) , acks=all ( ISR 전체에 복제되어야 OK )
실무에서는 보통 acks=all을 많이 쓰며, 0으로 갈수록 빨라지며 all 로 갈수록 안정적이게 된다.

데이터 유실 방지 조건으로는 아래와 같다. 해당 조건을 만족하는 경우 브로커가 하나 죽었을 경우에도 데이터가 안전하다.
replication factor ≥ 3
acks = all
min.insync.replicas ≥ 2
# min.insync.replicas 는 최소 복제 갯수로 replication 중 N개 이상이 복제되지 않을 경우 실패 처리한다.

- 장애 대응 (Fault Tolerance)
Controller Broker는 Leader의 상태를 감시하고 있다가 Leader에서 장애가 발생하면 ISR 중에서 하나를 Leader로 승격시킨다.
이후 Producer와 Consumer를 다시 연결시키고 메타데이터를 업데이트한다.
이 모든 과정은 자동으로 이루어진다.

만약 Follower에 장애가 발생했을 경우 ISR 에서 제외하기만 한다.
복제 수는 감소하긴 했지만 Leader가 살아있기 때문에 서비스는 계속 살아있게 된다.

- 클러스터 조정 ( Controller 역할 )
여러 브로커 중에서 한 브로커는 Controller 역할을 수행한다.
이때 Controller 역할을 수행하는 브로커는 Partition 리더 선출, 장애 감지, 재할당 등의 역할을 수행한다.

** 장애 감지
특정 Broker가 죽으면 해당 노드를 제거한다. Broker 장애는 ZooKeeper를 통해 신호를 받으며 모든 Broker는 Zookeeper에 등록되어 있다.
** Partition 리더 선출
특정 Broker에 장애가 발생하면 새로운 Partition Leader를 ISR 에서 찾아 등록한다.
** 재할당
Broker가 장애로 인해 제거되면 복제 구조가 깨지기 때문에 다른 Broker에 Replica를 구성한다.
그 외 새로운 Broker가 들어오면 데이터 균형을 위해 Partition의 일부를 새 Broker에 옮기고, 특정 Broker에 트래픽이 집중되면 해당 Partition을
다른 Broker로 이동시킨다.
# 기존 Replica에서 Leader만 변경하는 리더 선출과는 다르다. 재할당은 replica의 위치를 변경한다.
# 다만 재할당은 데이터 복사 비용, 네트워크 비용, 디스크 I/O 증가로 많은 비용으로 인해 자주 실행되지 않는다.

6. Cluster
Cluster는 확장성 ( Scale-Out ) 과 장애 대응을 위해 여러 브로커들을 하나로 묶은 구조이다.
즉, 여러 Broker를 하나로 묶어 고가용성 분산 시스템 처럼 동작하게 하는 구조이다.

클러스터 안에는 여러 개의 Broker와 1개의 Controller Broker 가 있다. 물론 Broker 안에는 다수의 Partition들이 존재한다.
Cluster는 Broker, Topic, Partition, Controller Broker 와 같은 Kafka 내부 리소스를 관리하고
Producer , Consumer, Consumer Group은 Cluster에 접속하는 외부 주체이다.

여러 개의 브로커들을 하나로 묶음으로 써 얻는 이점을 상세하게 나열하면 다음과 같다.
- 확장성 : 브로커를 추가하면 처리량이 증가하며, Partition들을 더 많이 분산할 수 있다.
- 장애 대응 : 특정 브로커가 죽어도 다른 브로커의 Replica ( Follower ) 로 대응할 수 있다. 이를 통해 서비스 중단 없이 계속 동작한다.
- 분산 처리 : Producer/Consumer의 요청을 여러 브로커가 나눠서 처리할 수 있다.
- 병렬 처리 : 여러 브로커에 Partition이 분산 저장되어있어 Consumer를 활용해 병렬 처리를 통해 처리량을 늘릴 수 있다.

그 외에 클러스터 내부에서는 어떤 브로커가 어떤 Partition을 가지고 있는지, Leader가 어디있는지 알 수 있으며 Controller 선출과 장애 대응을 한다.
즉, 한 클래스터 내에서 각각의 브로커들은 서로의 상태를 알고있다.

7. Consumer ( 소비자 )
메세지를 읽는 주체로 Pull을 통해 메세지를 가져온다.
이때 Offset을 기반으로 데이터를 읽으며, 어디까지 읽었는지 기록을 한다.

- Offset
메세지의 위치 ( index ) 를 말하며, Consumer가 직접 관리한다.
필요에 따라 offset 값을 되돌려서 재처리를 할 수 있다.

8. Consumer Group
Consumer Group은 하나의 메세지를 처리하기 위한 단위이며, 여러 개의 Consumer 들을 묶는 그룹이다.
Kafka는 같은 그룹 내에서 1개의 Message에 대해서 단 1개의 Consumer만 처리한다.
내부적으로는 1개의 Partition에 대해 같은 그룹에 있는 Consumer 중 단 1개만 할당된다.
그렇기 때문에 같은 그룹 내에서는 메세지를 중복처리 하지 않는다.

예를 들어 Partition이 3개가 있을 때, 한 Consumer Group 안에 Consumer가 3개가 있다면
각각의 Consumer들은 Partition을 나눠서 처리한다.
P1 -> C1
P2 -> C2
P3 -> C3
반대로 Consumer Group이 다르다면 같은 Message를 각각 읽는다.

만약 Consumer의 수가 Partition 수보다 많을 경우 남는 Consumer는 놀게 되며,
반대로 Partition의 수가 더 많을 경우 한 Consumer가 여러 Partition을 처리하기 때문에 부하가 증가한다.
그렇기 때문에 Consumer의 수는 Partition 의 갯수보다 같거나 작아야 한다.

- Consumer Group의 장점
여러 Consumer가 Partition을 나눠서 병렬 처리할 수 있으며, Consumer의 갯수를 늘려 수평 확장 ( scale-out ) 을 할 수 있다.
그리고 같은 그룹 내에선 중복 없이 단 1번만 처리되며,
Consumer가 죽으면 다른 Consumer가 Partition을 처리하는 리벨런싱 ( Rebalance ) 기능인 장애 대응 ( Failover ) 이 가능하다.

- Consumer Group의 단점
Consumer의 수가 많아도 Partition의 갯수가 적으면 병렬 처리에 한계가 있다. Partition이 3개면 Consumer가 아무리 많아도 실제로 3개의 Consumer만
일하게 된다.
그리고 Rebalance 로 인한 Partition 재분배 비용이 발생하는데, 일시적으로 메세지 소비가 멈춰져 처리에 지연이 발생할 수 있다.
또한 Kafka는 Partition 단위로만 순서를 보장하기 때문에 메세지 순서를 보장할 수 없다.
또한 Consumer에 장애가 발생하여 Offset Commit을 실패하게 되면 같은 메세지를 재 처리할 수 있어 멱등성 설계가 필요하다.

# 각 Consumer 들은 상태 공유가 어렵기 때문에 별도의 저장소를 활용해서 상태를 공유해야 한다.

정리하면 Consumer Group은 확장성과 안정성을 얻는 대신에 순서 보장과 Rebalance 비용을 감수해야하는 구조이다.

9. Zookeeper
Zookeeper는 분산 시스템에서 조율자 역할을 하는 시스템으로, Kafka 클러스터의 상태를 관리하고 리더를 선출한다.
Zookeeper는 내부적으로 파일에다가 상태를 저장하고, watcher 를 활용해 데이터 변경을 감지하였다.
과거에는 Kafka가 Zookeeper에 많이 의존했지만, 최신 Kafka 버전에서는 Zookeeper를 제거하는 KRaft 모드를 활용하고 있다.
KRaft 모드는 kafka가 Zookeeper 역할도 수행하는 것이다. ( KRaft는 Kafka Raft의 줄임말 )
Zookeeper를 제거하는 이유는 운영 복잡도가 증가하며, 장애 포인트가 발생할 가능성이 높기 때문이다.

Zookeeper의 주요 기능
- 브로커 관리
Broker의 상태를 확인하고 Broker를 추가하거나 삭제한다.

- Controller 설정 ( 리더 선출 )
장애 처리 및 Partition 리더를 할당하는 Controller Broker를 선출한다.

- 메타 데이터 저장
Topic 정보, Partition 정보 , Replica 위치 등 Kafka의 구조 정보를 저장한다.

- 장애감지 ( Failover )
Controller Broker가 죽으면 즉시 감지하여 자동으로 새로운 Controller Broker를 선출한다.
이후 해당 Controller 가 전체 Cluster 메타데이터 관리를 이어 받는다.

# Broker가 죽어서 Partition Leader 와 함께 기능이 정지될 경우 ISR 에서 새로운 Leader를 호출하는 것은 Controller Broker가 수행한다.

[ Replica 레플리카 ]
Replica는 Partition을 여러 Broker에 복제하여, 고가용성의 구조로 만드는 개념이다.
# 고가용성이란 시스템에 장애가 발생하여도 서비스 중단 없이 지속적으로 운영되는 구조를 말한다.
Kafka는 Partition 단위로 저장된 데이터를 여러 Broker에 복제해 둔다.
ex )
Broker1 - Partition 1 ( Leader )
Broker2 - Partition 1 ( Follower )
Broker3 - Partition 1 ( Follower )
이를 통해 Broker1 이 죽어도 Broker2 혹은 Broker3 이 대신 처리하여 서비스 중단 없이 지속되는 고가용성의 특징을 가진다.

- Leader / Follower 구조
레플리카는 Partition 복제뿐만 아니라, Leader Replica 와 Follower Replica 도 존재한다.

Leader Replica는 데이터 읽기 및 쓰기를 진행하고 Producer/Consumer와 통신을 한다.
그리고 Follower Replica는 평소에 Leader 데이터를 복제만 진행하고, Leader가 죽었을 경우 Follower 중 하나가 Leader로 승격한다.
이때 Leader 승격은 ISR 내부에서만 발생하며, ISR 내부에서는 Leader와 동기화가 잘 된 Follower Replica 들이 존재한다.

# ISR은 In-Sync Replica 의 약자이다.
# 동기화가 안된 Follower Replica 는 제외된다.

- replication factor
Kafka는 전체 Replica 갯수를 설정할 수 있다.
값이 높을 수록 안정성이 높아지지만, 저장 공간 비용과 네트워크 비용또한 증가한다는 단점을 가지고 있다.
Leader의 개수는 1개 고정이며, 나머지는 모두 Follower 로 배정된다.
예를 들어 replication factor 의 값이 3일 경우 1개의 Leader와 2개의 Follower 를 가지게 된다.

replication factor가 1인 경우 Follower Replica는 없으며, 장애가 발생하면 복구가 불가능하다.
일반적으로 3을 가장 많이 쓴다.

[ Kafka의 특징 ]
1. 디스크 기반 고성능 조회
Kafka는 디스크 로드를 통해 데이터를 읽지만 Random I/O 가 아닌 Sequential I/O 방식으로 데이터를 조회하여 속도가 빠르다.

2. 부분적 순서 보장
같은 Partition 내부에서만 순서를 보장한다.

3. 높은 확장성
Partition을 늘리면 처리량이 증가한다.

4. 다양한 장애 대응
Replication ( 복제 ) 이나 Leader Election 방식을 통해 장애를 극복할 수 있다.

5. Pull 기반
Consumer는 속도를 조절할 수 있다.
