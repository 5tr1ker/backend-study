[ 전통 MQ 란? ]
전통 MQ ( Message Queue ) 는 작업 처리용 큐이다.
전통 MQ에는 대표적으로 RabbitMQ 와 ActiveMQ , ArtemisMQ 등이 있으며, RabbitMQ 를 그중에서 가장 많이 사용한다.

[ 전통 MQ의 구성 요소 - RabbitMQ 기준 ]
Broker
Producer → Exchange → Queue → Consumer

- Producer ( 생산자 )
메세지를 생성해서 MQ로 보내는 주체이다.

- Consumer ( 소비자 )
큐에서 메세지를 가져가는 주체이며, 처리 완료 시 ACK 신호를 보낸다.
여러 개의 Consumer 를 연결하여 병렬 처리를 할 수 있다.
다만, 같은 Queue에 여러 개의 Consumer가 연결된 경우에는 하나의 메세지는 1개의 Consumer만 갖는 경쟁 구조가 된다.
즉, 여러 개의 Consumer 가 한 큐에 붙어있다 하더라도, 하나의 메세지는 한 Consumer 만 처리한다.

- Broker ( MQ 서버 )
메세지를 저장, 전달, 관리하는 핵심 시스템이다.
대표적으로 메세지를 저장하고, 라우팅을 하며, Consumer에게 메세지를 전달하는 역할을 한다.
Broker는 메세지에 대해 상태 관리를 하며, Consumer에게 처리가 완료되었다는 신호인 ACK 신호를 받으면 해당 메세지를 삭제하고,
실패 신호인 NACK 신호를 받으면 재처리하거나 DLQ에 보낸다.

메세지가 가질 수 있는 상태는 다음과 같다.
1. Ready : 아무도 가져가지 않은 상태
2. In_Flight : Consumer에게 메세지가 전달됐지만 ACK 신호를 받지 못한 상태 ( 이때 어떤 Consumer 가 처리중인지 기록 )
    ㄴ 이때 Consumer 가 ACK 신호를 보내지 못하고 죽을 경우 다시 Ready 상태로 갈 수 있다. 이때 메세지가 중복 전달되는 At-Least-Once 가 발생한다.
3. Acked : Consumer에게 ACK 신호를 받은 상태
4. Expired : TTL 이 지난 메세지
    ㄴ TTL 은 Producer가 정할 수 있으며, 요청된 메세지가 TTL 기간동안 처리되지 못할 경우 삭제되거나 DLQ로 이동된다. ( 기본 값은 삭제 )

Broker는 ACK 신호를 받으면 해당 메세지를 Queue에서 제거하고, 디스크 제거 대상에 등록한다.
메세지는 ACK 신호를 받은 뒤에 삭제되지만 내부적으로는 삭제 표시만 남겨두고 나중에 한번에 삭제할 수 있다.

Broker는 Consumer에게 메세지를 Push 기반으로 전송하는데, Consumer가 연결될 경우 메세지를 하나씩 건네준다.
이후 Consumer가 ACK 신호를 주면 다음 메세지를 전송한다. 이때 메세지는 1개씩 전송한다.
# 1개씩 전송하면 속도가 느리기 때문에 N개의 메세지를 한번에 보낼 수 있는 Prefetch 방식이 있다. 다만 ACK 관리가 복잡해진다.

- Queue ( 큐 )
실제 메세지가 쌓이는 공간으로 작업 대기열이다.
FIFO 의 구조를 가지고 있으며, Consumer가 메세지를 가져갈 때 까지 대기한다.
Queue는 작업 종류별로 다수의 큐를 배치할 수 있지만, 순서 보장은 동일한 Queue일 경우에만 순서를 보장한다.

Queue는 Broker에 의해 관리되고 있으며, Queue에 들어간 데이터는 메모리나 디스크에 보관한다. ( 둘다 저장할 수 있다. )
메모리를 사용할 경우 데이터 저장 속도가 빠르지만 Broker가 죽을 경우 메세지가 유실된다는 단점이 가지고 있다.
그래서 로그나 임시 작업 같은 작업에 사용한다.

디스크를 사용하는 경우 장애가 발생해도 복구할 수 있는 특징을 가지고 있으며, 대부분 환경에서는 디스크에 데이터를 저장한다.
이때 디스크에 데이터를 먼저 저장하고 Queue에 등록한 다음 Consumer에게 전달한다.

- Exchange ( 라우터 )
Producer가 보낸 메세지를 Exchange ( 라우터 ) 가 받아서 어떤 Queue로 보낼지 결정한다.
Exchange는 메세지의 routing key와 Queue의 Binding key를 기준으로 메세지를 라우팅하며,
라우팅의 대표적인 방법으로는 direct , topic , fanout 이 있다.

1. Direct Exchange
가장 단순한 방식으로 routing key 와 일치하는 Queue로 메세지가 전달된다.
1:1로 라우팅이 되며 특정 Queue를 지정하여 메세지를 전달할 때 활용된다.

특정 서비스에 대한 작업을 처리할 때 활용된다.
예를 들어 사용자를 제거하는 작업으로 routing key 가 user.deleted 일 때, Queue의 Binding 설정이 user.deleted 이어야 라우팅된다.

2. Topic Exchange
패턴 기반 라우팅으로 와일드 카드 ( * , # ) 를 활용해서 여러 Queue에 전달할 수 있다.
# * 는 단어 1개를 의미하고, # 는 여러 단어를 의미한다.

예를 들어 routing key 가 user.created 일 때, Queue가 user.* 이거나 user.# 이면 해당 Queue는 메세지를 받아 처리한다.
user.login.fail 일 때 user.* 는 받지 못하고 user.# 는 메세지를 받는다.
user 일 때 user.* 는 받지 못하고 user.# 는 메세지를 받는다.
즉, * 는 해당 자릿수와 일치해야 하고, # 는 0개 혹은 다수의 문자열을 받는다.

Topic Exchange는 이벤트 기반 아키텍쳐에 활용되며, Kafka의 Topic 이름 전략과 유사하다.
그리고 MSA 구조에서 이벤트를 전달하기 위해 활용되기도 한다.

3. Fanout Exchange
routing key와 관계 없이 연결된 모든 Queue에게 메세지를 전달한다.

이는 Pub/Sub 구조 , 실시간 알림 , 로그 전파에 활용된다.

- Message ( 메세지 )
실제 전달되는 데이터로 Body 와 Header , Routing Key 로 구성이 되어 있다.
Routing Key는 Exchange 에서 활용된다.

- Message 와 Queue의 저장 위치 / 영속화
Queue와 Message는 기본적으로 메모리에만 저장이 되며, 옵션을 통해 디스크 영속화 ( persistence ) 를 활성화 하면
메모리 뿐만 아니라 디스크에 파일 형태로 저장을 한다.
정보를 디스크에 저장하는 이유는 서버 장애로 인한 데이터 유실을 방지하기 위해서이다.

영속화에 대한 개념은 크게 Queue Durable와 Message Persistent로 나뉜다.
Queue Durable : Queue 에 대한 메타데이터 정보를 디스크에 저장하는 것을 말한다.
Message Persistent : Message 데이터를 디스크에 저장하는 것을 말한다.

Broker는 Queue에 대한 메타데이터 정보인 Queue 이름, Consumer 연결 정보, Binding 정보, Durable 여부, 설정 정보들을
기본적으로 메모리에만 저장하고, Queue Durable 설정 시 내부 DB 혹은 메타데이터 파일로 저장된다.
기본 값은 durable = false 이지만, true 값으로 변경을 통해 활성화 할 수 있다.
영속화 설정이 되어 있지 않으면, Broker 재 시작시 Queue 가 삭제된다.

Message는 기본적으로 메모리에만 적재되고, Message Persistent 설정 시 디스크에 파일로 저장한다.
예를 들어 Broker는 Producer에게 Message를 받으면 메모리에 적재하고, 설정 시 디스크에 파일로 저장한다.

안정성을 위해서 Queue Durable 과 Message Persistent 를 활성화하는 것이 좋지만, Disk I/O 로 인해 성능이 저하된다는 단점이 있다.
만약, 실시간 로그나 모니터링과 같이 실시간성이 중요한 경우 non-persistent 를 사용해도 무방하다.

# 다만 영속성 설정을 한다고 해서 바로 디스크에 반영되는 것은 아니다. 데이터들을 메모리에 잠깐 쌓아둔 후 배치를 통해 flush 하기도 한다.

- ACK / NACK
메세지의 처리 결과이며 Consumer가 Broker에게 알려준다.
ACK는 처리 완료의 신호로 Broker는 해당 메세지를 삭제한다.
NACK는 처리 실패의 신호로 재전송을 하거나 DLQ 로 보낸다.

[ DLQ ( Dead Letter Queue ) ]
DLQ는 처리 실패한 메세지의 보관소이며 처리 실패한 메세지를 분석할 용도로 사용한다.
DLQ는 일반 Queue와는 다른 별도의 Queue이며, 운영자는 Queue안에 메세지를 Peek/Browse 방식으로 조회할 수 있다.
# Peek/Browse 방식을 통해 큐 안에 몇번째 데이터를 조회할 수 있다.

DLQ 메세지는 기본적으로 즉시 삭제되지 않으며, 보관 정책을 통해 일정 기간 유지시킬 수 있다. ( 7일 후 삭제 )
DLQ 메세지는 다시 Consumer에게 전달할 수 있으며, Consumer가 ACK 신호를 전송하면 해당 메세지는 삭제된다.
혹은 DLQ Size 를 제한하여 특정 용량을 넘을 경우 오래된 메세지부터 삭제하게 할 수 있다.

# DLQ는 보통 Consumer에서 여러번 실패한 메세지, retry 초과, deserialize 실패 등의 사유로 메세지가 들어온다.

[ Broker ]
대표적인 MQ 중 RabbitMQ와 ActiveMQ , IBM MQ들의 Broker는 다음과 같은 역할을 수행한다.
메세지 저장, 큐 상태 관리, Consumer 전달, Ack관리, 재전송/삭제, 장애 복구

우선 Producer가 Broker로 메세지를 전달하면 Broker는 내부 Queue에 메세지를 저장한다.

[ Queue 패턴 ]

- Work Queue
1개의 Queue에 N개의 Consumer가 붙는 구조이다.
작업 분산 처리를 위해 구성한다.

- Routing 조건별 분기
Exchange -> QueueA 혹은 QueueB
Binding 규칙을 통해 메세지의 규칙에 따라 특정 Queue에 메세지를 전달한다.

- Pub/Sub ( Fanout )
Producer -> 여러개의 Queue
Producer는 여러 개의 Queue에 같은 메세지를 발행할 수 있으며, 이를 통해 여러 Consumer 가 동일 메세지를 처리하게 할 수 있다.

[ 메세지 분배 방법 ]
하나의 큐에 대해 다수의 Consumer가 연결되어 있는 경우 하나의 메세지에 대해 하나의 Consumer만 처리하게 된다.
이때 메세지를 분배하기 위한 방법으로는 Round-Robin 방법과 Prefetch 방법이 있다.

- Round-Robin
여러 Consumer 들을 돌아가면서 순서대로 1개씩 분배한다.

- Prefetch
속도 기반 분배로 빨리 처리되는 Consumer에게 더 많은 메세지가 할당된다.

- Broadcast ( Fan-Out )

[ 전통 MQ 특징 ]
- Push 기반 모델
Broker가 Consumer에게 메세지를 밀어 넣는다.
메세지가 빠르게 전달되지만, Consumer에 부하가 발생할 수 있다.

- 메세지 소비 후 삭제
Consumer가 소비된 메세지는 삭제된다.
그렇기 때문에 한번 소비된 메세지는 재처리될 수 없다.

# Kafka는 메세지를 로그에 저장해두고 일정 기간 유지한다. ( Retention )

- 상태 기반
메세지가 처리됐는지에 대한 상태를 관리한다.
상태는 ACK, NACK , 재전송 등이 있다.

- 메세지 순서 보장
단일 Queue 기준으로 순서를 보장한다.

- 확장성 제한
전통 MQ는 Broker 중심 구조이기 때문에 수평 확장이 어렵다.
