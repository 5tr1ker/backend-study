[ MQ의 종류 ]
전통 MQ 에는 여러 오픈소스들이 존재한다. RabbitMQ와 ActiveMQ ( classic , Artemis ) , IBM MQ , TIBCO MQ , Amazon SQS 등이 존재하지만,
오픈소스 MQ에서는 RabbitMQ 와 ArtemisMQ 가 대표적으로 많이 사용한다.
ArtemisMQ는 ActiveMQ의 개선 버전으로 기존 성능 문제를 개선 시킨 차세대 MQ이다.

[ RabbitMQ ]
가장 많이 사용되고 있는 MQ 중 하나이다.
업무 처리에 있어서 메세지 신뢰성이 높으며, 간편한 UI를 제공하여 MSA 아키텍쳐에서 쉽게 모니터링을 할 수 있다.

RabbitMQ는 AMQP 프로토콜을 기반으로 Exchange -> Queue -> Consumer 구조로 구성된다.

- AMQP 프로토콜
AMQP ( Advanced Message Queue Protocol ) 은 메세지 브로커와 클라이언트가 통신하기 위한 표준 프로토콜이다.
즉, Producer 가 Message를 보내고 Consumer가 메세지를 받기 위한 규칙이다.

AMQP 프로토콜은 메세지 포맷, 연결 방식, ACK 처리, Queue 생성, Exchange 생성, Routing 등을 표준화 한다.
여기서 RabbitMQ는 AMQP 프로토콜을 기반으로 설계되었다. 그와는 달리 Kafka는 자체 프로토콜을 가지고 있으며,
ArtemisMQ는 자체 Core Protocol을 기반으로 설계되었지만 AMQP 를 포함한 다양한 프로토콜을 지원한다.

- 다양한 라우팅
RabbitMQ는 다양한 Exchange 를 제공하여 메세지를 원하는 Queue에 전달하거나, 여러 Queue에 복사하여 전달할 수 있다.
Exchange 는 Direct , Topic , Fanout , Headers 방식을 제공하고 있다.

Exchange는 Queue와 Producer 사이에 위치하여, Producer가 전송한 Message를 Exchange가 먼저 받고, Exchange가 Queue에게 메세지를 전달한다.
ArtemisMQ는 Exchange라는 개념이 없으며 유사한 역할을 하는 Address가 제공된다.
ArtemisMQ에서 Producer가 Address로 메세지를 전송하면, Broker는 해당 Address가 바인딩된 Queue들을 찾아 메세지를 전달한다.
이때 동일한 Address 가 바인딩된 Queue가 다수가 있다면 Routing Type에 따라 1개만 메세지를 전달할지 ( ANYCAST ) , 전부 전달할지 ( MULTICAST )
설정할 수 있다.

RabbitMQ는 Producer -> Exchange -> Queue 구조로 Exchange 중심의 메세지 라우팅이라면,
ArtemisMQ는 Producer -> Address -> Queue 구조인 Address 중심의 메세지 라우팅 구조가 된다.

- 운영을 위한 UI 제공
큐 상태, Consumer 상태, 메세지 수, DLQ, Cluster 상태 등 현재 MQ 서버 내의 다양한 정보들을 UI로 간단하게 볼 수 있다.
물론 UI를 사용하기 위해 플러그인을 추가해야 한다.

- 다양한 플러그인 지원
RabbitMQ는 다양한 플러그인을 지원하여 필요한 기능을 추가할 수 있다.
이는 필요한 기능만 추가하여 활성화할 수 있기 때문에 가볍다는 장점이 있다.

Management Plugin ( 웹 UI ) , Federation Plugin ( Broker간 메세지 연동 ) , Shovel Plugin ( 메세지 복사 ) ,
MQTT Plugin ( MQTT Broker ) , STOMP Plugin ( WebSocket , STOMP 클라이언트 연동 ) 등 다양한 플러그인을 제공한다.

그와는 반대로 ArtemisMQ는 엔터프라이즈 기능이 기본 내장되어 있어 초기 기능을 활용하는데 난이도가 있는 편이다.

- 메세지 신뢰성
RabbitMQ는 ACK, DLQ, Queue Durable , Persistent Message , Quorum Queue Replication 등을 제공하여 메세지의 유실을 최소화 한다.
특정 Broker나 Queue에 장애가 발생하여도 메세지를 복구할 수 있으며, ACK 및 영속성 설정을 통해 At-Least-Once 를 보장한다.

사실 해당 특징은 RabbitMQ 뿐만이 아니라 ArtemisMQ 역시도 지원한다.

[ ArtemisMQ ]
Apache에서 제공하는 차세대 ActiveMQ Broker이다.
ArtemisMQ는 ActiveMQ Classic 의 성능 문제를 개선 시킨 후속 버전이다.

ArtemisMQ는 운영과 학습 난이도가 RabbitMQ 보다 높지만, Java 와 호환성이 좋고 대규모 환경 및 성능이 우수하다는 특징을 가지고 있다.

- 고성능
ArtemisMQ는 Journal 이라는 전용 저장소에 메세지를 빠르게 적재한다.
Journal은 append-only 구조이기 때문에 새로 들어온 메세지를 파일 끝에 추가하기만 한다.

RabbitMQ 역시 영속 메세지를 디스크에 저장하지만 ArtemisMQ는 Journal 기반 저장소를 활용하기 때문에 디스크에 쓰는 속도는 RabbitMQ 보다 빠르다.
그 이유는 Sequential Write ( Sequential I/O ) 에 더 많이 최적화 되어 있기 때문이다.
그리고 Asynchronous I/O ( 비동기 I/O ) 를 활용하여 데이터 쓰기 요청 후 기다리지 않고 하고 바로 다음 작업으로 넘어가고,
Batch Flush 를 통해 Flush 를 최소화하여 Disk I/O 를 최소화하여 성능을 향상시켰다.
이러한 이유로 대량의 영속 메세지를 처리하는 환경에서 RabbitMQ 와의 성능 차이가 더 부곽된다.

# RabbitMQ 도 Sequential I/O 를 사용하지만, ArtemisMQ는 이를 더욱 최적화시켰다.
# 단순 라우팅 환경에서는 ArtemisMQ와 RabbitMQ는 크게 차이가 발생하지 않을 수 있다.

- 트랜잭션 지원
Producer는 메세지 전송 시 commit 시점에 메세지를 확정한다.
이때 메세지를 전송하는 도중에 오류가 발생하면 rollback 을 통해 메세지 전송을 취소할 수 있다.
그렇게 되면 이미 보낸 메세지도 전송 취소가 되어 어떠한 메세지도 저장되지 않는다.

Consumer의 경우 메세지를 읽고 처리한 후 ACK 신호를 보내면 메세지가 삭제되지만,
처리 과정에서 오류가 발생하면 rollback을 통해 해당 메세지는 삭제되지 않으며, Broker 정책에 따라 메세지를 재전송하게 된다.

이를 통해 트랜잭션 범위에 있는 모든 작업은 수행되거나 수행되지 않는 원자성의 특징을 얻을 수 있게 된다.

# 트랜잭션은 ArtemisMQ 뿐만이 아니라 전통 MQ의 특징이다.

이때 DB 작업과 MQ의 작업을 한 트랜잭션 범위를 묶고 싶다고 하면 XA Transaction 기능을 활용하면 된다.
다만, 속도가 느리고 복잡하기 때문에 대안으로 Outbox Pattern 을 사용한다.

- Outbox Pattern
Outbox Pattern 는 DB에 이벤트를 저장하고 별도의 프로세스가 MQ로 전달하는 패턴이다.
이때 테이블에 이벤트에 대한 메세지를 저장하고, CDC가 테이블의 변동을 감지하여 이벤트를 전달한다.

해당 패턴은 MSA 구조에서 분리된 서비스간에 Queue를 활용하여 통신할 때, DB와 메세지 간의 일관성을 유지하기 위해 사용될 수 있다.
여기에 Outbox Pattern 은 구현이 단순하고 안정적이기 때문에 많이 활용된다.

# CDC ( Debezium ) 방식 말고도 Polling 방식이 있지만, CDC를 많이 사용한다.

이를 활용한 사례로 주문 이벤트가 들어왔다면 하나의 트랜잭션에서 주문 테이블과 outbox 테이블에 이벤트를 저장하고,
CDC 혹은 별도의 Publisher가 이벤트를 MQ로 발행한다.
# CDC ( Debezium + Kafka ) 방식과 Polling 방식이 있다.

이때 outbox 테이블에는 메세지, 상태 값이 포함되어 있을 수 있다.
# id / event_type / payload / status / created_at / published_at

- JMS 표준 지원
JMS 표준을 지원하기 때문에 Java 기반 환경에서 높은 호환성을 가진다.
JMS ( Java Message Service ) 는 Java 가 메세지 큐를 사용할 때 지켜야 하는 표준 인터페이스이다.

JDBC처럼 JMS 표준을 통해 MQ 가 변경되어도 소스 코드의 변경을 최소화 한다는 장점을 가지고 있다.
JMS 는 Producer 와 Consumer, Message, Connection , Session , Topic , Queue 등의 표준을 정의하고 있으며, 구현은 MQ 벤더가 관리하고 있어
개발자는 용도에 맞는 메서드 ( JMS API ) 만 활용하면 된다.

RabbitMQ는 JMS 기반이 아닌 AMQP 기반의 메세지 브로커이며, 기본적으로 RabbitMQ 전용 API를 사용한다.
다만, JMS 호환 라이브러리를 활용하여 JMS 방식을 적용할 수 있다.

# JMS는 프로토콜이 아닌 Java API 표준이다.

- 다양한 프로토콜 지원 ( 멀티 프로토콜 지원 )
ArtemisMQ는 다양한 프로토콜을 지원하고 있기 때문에 다양한 시스템들을 하나의 Broker에 연결할 수 있다.
ArtemisMQ가 제공하는 프로토콜들은 AMQP, MQTT , STOMP , OpenWire 등이 있다.

1. Core Protocol
ArtemisMQ의 기본 프로토콜로 다른 프로토콜보다 더 나은 성능을 가지고 있으며 Artemis 의 기능을 완전히 활용할 수 있다.
다만, Artemis에 종속된다는 단점을 가지고 있다.

2. AMQP 1.0
MQ 업계에서 많이 사용하는 프로토콜 중 하나로 언어 독립적이라는 특징을 가지고 있다.
RabbitMQ와 ArtemisMQ , Azure Service Bus 등 해당 프로토콜을 사용하는 곳이 많다.
그래서 같은 프로토콜을 사용하는 서로 다른 언어의 어플리케이션에서 동일 방식으로 Broker와 통신할 수 있다.

3. MQTT
IoT용 프로토콜로 가볍고 전력을 적게 사용하며, 모바일, IoT 기기에 최적화되어 있다.

4. STOMP ( Simple Text Oriented Messaging Protocol )
사람이 읽을 수 있는 텍스트 기반의 프로토콜이다.
웹 소켓이나 브라우저, 간단한 테스트에 사용된다.

5. OpenWire
ActiveMQ Classic 호환용이다.

- 대규모 클러스터링
Broker Cluster와 Replication, Fail Over , Broker-to-Broker 구성을 지원하기 때문에
요청량이 많은 곳에서도 안정적으로 운영될 수 있다.

- 대규모 엔터프라이즈 시스템에 적합
대규모 엔터프라이즈 시스템은 다음과 같은 특징을 가지고 있다.
1. 24시간 운영
2. 높은 신뢰성과 안정성이 중요
3. 장애 발생 시 영향 범위가 큼
4. 데이터 유실을 허용하지 않음
5. 수많은 업무 시스템이 연동됨

ArtemisMQ는 메세지 영속화와 트랜잭션 지원, 높은 처리량 ( 비동기 I/O , Journal 기반 저장 , 효율적인 메모리 관리 ) , 다양한 프로토콜 지원,
클러스터링과 HA ( High Availability ) 을 지원한다.
그렇기 때문에 메세지 유실이 거의 허용되지 않는 기업용 업무 환경에서 많이 활용된다.

- ActiveMQ의 후속 버전
ArtemisMQ는 ActiveMQ Classic 의 후속 버전으로 편의 기능 및 성능이 대폭 향상되었다.
ArtemisMQ는 HornetQ 기반으로 재설계되어 성능과 확장성, 안정성이 크게 개선되었다.

1. 성능 개선
ActiveMQ는 kahaDB를 사용하여 메세지를 저장하였지만, 대량의 영속 메세지를 처리하는 환경에서는 한계가 있었다.
이에 ArtemisMQ 는 Journal 저장소를 활용하여 영속 메세지 처리 성능을 향상시켰다.

2. 메모리 사용 개선
ActiveMQ는 메세지가 많아지면 메모리 사용량이 급격하게 늘어나는 현상이 있었지만,
ArtemisMQ 에서는 최적화가 되어 대량 메세지에서 안정적으로 처리되게 개선되었다.

3. 클러스터링 개선
ActiveMQ 에서는 클러스터 구성이 복잡했지만, ArtemisMQ에서는 확장성이 좋아졌다.

4. 멀티 프로토콜 강화
ActiveMQ 도 기본적으로 여러 프로토콜을 지원했지만, ArtemisMQ 는 멀티 프로토콜 Broker 를 목표로 설계되었기 때문에
다양한 프로토콜들을 더욱 효율적으로 지원한다.

5. JMS 2.0 지원
ActiveMQ는 오래된 JMS 버전이지만, ArtemisMQ는 JMS 2.0 및 Jakarta Messaging 등 최신 Java 환경을 지원한다.
