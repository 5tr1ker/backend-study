[ Queue 란 무엇인가 ]
Queue 는 일반적으로 FIFO ( First In First Out ) 의 구조로 설계된 큐이다.
FIFO의 특징 답게 먼저 들어온 요청이 먼저 처리되지만, 서버가 분산된 환경에서는 순서가 보장되지 않을 수 있다.
웹 환경에서는 서버가 요청을 처리하는 과정에서 queue를 활용하여 비동기적으로 요청을 처리한다.
이를 통해 요청을 바로 처리하는 것이 아닌 Queue에 안전하게 쌓아두고, 천천히 처리하는 구조로 활용된다.

- 웹 환경에서 Queue를 두는 이유
웹 환경에서 Queue를 두는 큰 이유는 부하 분산때문이다.
클라이언트에서 갑자기 요청이 몰리게 될 경우, 서버는 이를 처리하다가 부하가 걸려 서버가 다운될 수 있다.
이때 서버는 Queue에 요청을 넣으면, worker는 Queue에 있는 요청을 하나씩 꺼내어 처리하여 요청 폭주로 부터 서버가 안전해질 수 있다.
즉, Queue는 버퍼 역할도 할 수 있다.

또 다른 장점으로는 비동기 처리이다.
로그 처리나, 결제에 대한 문자 발송, 회원가입으로 인한 메일 확인 발송같은 경우는 즉시 처리되지 않아도 된다.
이러한 우선 순위가 낮은 작업들은 Queue 에 넣어두고 사용자에게는 필요한 정보만 빠르게 제공할 수 있다.

그리고 세번째 장점으로는 서버 분리이다.
서버가 분리되어 있어 시스템을 독립적으로 확장할 수 있으며,
ACK 설정 및 영속성 설정 등이 잘 되어 있으면 서버에 장애가 발생하여도 큐에는 데이터가 남아 있기 때문에 데이터가 유실되는 상황을 해소할 수 있다.

[ Queue의 기본 개념 ]
Producer : Queue에 메세지를 넣는 주체
Consumer : Queue에서 메세지를 가져와 소비하는 주체
Message : Queue에 들어가는 메세지로 JSON 데이터가 일반적이다.
ACK ( Acknowledgement ) : 처리 완료 신호로 실패한 메세지는 다시 Queue로 들어간다.
Retry , Dead Letter Queue : 계속 실패하는 메세지가 보관되는 장소

실무에서는 Kafka 와 MQ 를 가장 많이 사용한다.
물론 다른 Queue도 더 있지만, 위의 2개의 Queue를 활용하면 대부분의 Queue 기반 아키텍쳐를 구성할 수 있다.

[ Queue를 사용하면 좋을 때 ]
요청이 순간적으로 몰리거나, 시스템을 분리할 때, 실패 재 처리가 필요하거나 바로 처리할 필요가 없을 때 Queue를 활용하면
안정적인 시스템을 만들 수 있다.

[ Kafka 와 메세지 큐 ]
Kafka 와 MQ는 둘다 Queue 를 활용한 시스템이지만 철학이 다르다.

- 데이터 스트림 ( kafka )
Kafka는 스트리밍 플랫폼처럼 데이터를 쌓아놓고 읽는 로그 시스템이다.
Kafka는 발행 ( Publish ), 구독 ( Subscribe ) 기반으로 로그를 저장하며 메세지를 소비해도 제거되지 않는다.
파티션 기반이기 때문에 수평 확장이 용이하며, 고성능이며 대용량 처리에 특화되어 있다.
그리고 Consumer는 같은 메세지를 독립적으로 읽는다. 쉽게 말하면 Producer가 발행한 메세지에 대해 여러 Consumer 들이 각각 읽는다.

Kafka는 주로 이벤트 기반 아키텍쳐, 로그/데이터 수집, 실시간 처리나 트래픽이 많은 경우에 활용된다.
그렇기 때문에 트래픽이 많으며, 같은 메세지를 여러 시스템에서 활용되어야 하고 같은 메세지를 여러번 활용해야 할 때 Kafka를 선택하면 된다.

- 메세지 큐 ( MQ )
메세지 큐는 작업 처리용 큐이며, Kafka와는 다르게 소비한 메세지는 삭제된다.
Producer 와 Consumer 는 1:1 이거나 경쟁 소비를 한다.
그리고 ACK와 DLQ, Retry를 통해 메세지 전달이 보장이 되며, 구조가 단순하다.

MQ는 트랜잭션 안정성이 중요하고, 비동기 작업을 처리하며 요청 처리 순서가 중요할 때 활용된다.
그렇기 때문에 작업은 한번만 처리되어야 하고, 실패하면 재시도해야 하며 업무 로직을 처리하는 경우에 MQ를 사용하면 된다. ( 결제 , 문자 전송 )
