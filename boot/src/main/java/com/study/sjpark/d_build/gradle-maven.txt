[ Maven 과 Gradle 의 차이 ]
Maven 과 Gradle은 모두 빌드 도구이지만, 설계 철학과 동작 방식이 다르다.
기존 Spring 프로젝트에서는 Maven 을 많이 사용했지만, 최근 프로젝트에서는 Gradle 을 많이 사용한다.

- Maven
Maven 은 대기업이나 SI 환경 등 기존 시스템과의 호환성을 위해 Maven 을 주로 사용한다.
Maven 은 정해진 방식대로 안정적이고 일관적인 빌드를 추구하고 있다.

1. XML 기반
Maven 은 XML을 사용한다.
구조가 정해져 있어 읽기 쉽지만 코드가 길어진다는 단점을 가지고 있다.

2. 선언형
Maven 은 선언형으로써 조건문이나 반복문과 같은 로직을 작성할 수 없다.
즉, 정해진 규칙 안에서만 동작할 수 있다.

이는 읽기가 쉽고 설정이 단순해진다는 장점을 가지고 있지만,
복잡한 로직 등 사용자 정의 작업을 수행할 수 없어 유연성이 떨어진다는 단점을 가지고 있다.

3. 낮은 빌드 속도
Maven은 빌드를 할 때 마다 clean > Compile > Test > Package > Install 의 순서를 항상 거치게 된다.
그렇기 때문에 프로젝트의 규모가 커질수록 빌드의 속도가 느려진다.

만약 1000개의 Java 파일 중에서 1개만 수정되었다 하더라도, 전체를 지우고 다시 빌드한다.

4. 순차 빌드
만약 여러 모듈들을 동시에 빌드를 할 경우 순차적으로 빌드를 수행한다.
즉, 병렬로 빌드를 수행할 수 없다.

# Maven 에서 모듈은 하나의 프로젝트를 하위의 여러 프로젝트 ( Sub Project ) 로 나눈 것을 의미한다.

5. 쉬운 학습 난이도
Maven은 dependency 와 plugin, repository 등 비교적 학습할 내용이 적지만,
Gradle은 프로그래밍 언어이기 때문에 변수나 문법 등을 학습해야 하기에 초반에는 난이도가 있다.

6. 안정적 ( Stable ) 이고 재현 가능한 빌드 ( Reproducible Build )
Maven은 Convention over Configuration 철학을 따르기 때문에 안정적 ( Stable ) 인 빌드가 가능하여 프로젝트마다 빌드 방식이 크게 달라지지 않는다.
그렇기 때문에 어떠한 프로젝트라도 구조가 비슷하다는 특징을 가지고 있다.

재현 가능한 빌드 ( Reproducible Build ) 는 언제 어디서든 빌드를 수행한 결과가 동일하다는 것을 말한다.
그 이유는 설정 파일 ( pom.xml ) 에 프로젝트에 대한 모든 정보를 기입하기 때문이다.
예를 들어 라이브러리와 버전, 플러그인과 패키징 방식 등의 프로젝트 설정은 모두 pom.xml 에 기입한다.

7. Convention over Configuration ( 설정보다 규칙을 우선 )
복잡한 설정보다 정해진 규칙을 따른다는 Maven의 철학이다.
개발자는 별도의 설정 없이 Maven이 정해놓은 표준을 따른다면,
개발자마다 빌드 방식이 달라질 가능성이 줄어들어 안정적이고 재현 가능한 빌드를 제공할 수 있다.

예를 들어 Maven은 프로젝트 구조를 미리 정의해 두었기 때문에 개발자는 java 소스와 리소스, 테스트 코드를 정해진 경로에 배치하기만 하면된다.
또한 패키징 방식 ( jar 혹은 war ) 만 정의하면, Maven이 미리 정의해둔 빌드 라이프사이클에 따라 빌드를 자동으로 수행하므로
개발자는 빌드 과정을 직접 설정할 필요가 없게 된다.

만약 이러한 규칙이 없다면 프로젝트마다 경로를 설정해야 하므로, 프로젝트 마다 구조와 빌드 방식이 달라지게 될 것이다.

- Gradle
Gradle은 프로젝트가 커져도 빌드 속도가 빠르며 설정의 유연성에서 큰 장점을 가지고 있다.
또한 필요에 따라 빌드 과정을 프로그래밍으로 자유롭게 확장하는 철학을 가지고 있다.

1. Groovy / Kotlin DSL 기반
Gradle은 Groovy / Kotlin DSL 을 사용하기 때문에
선언형과 프로그래밍 요소를 함께 사용할 수 있어 표현력이 높다는 장점을 가지고 있다.

2. 프로그래밍이 가능한 빌드 스크립트
Gradle 은 빌드 스크립트 자체가 프로그래밍 언어이기 때문에 조건문이나 반복문을 사용할 수 있다.
그렇기 때문에 빌드 과정을 코드처럼 작성할 수 있다.

이를 활용하여 운영 환경과 개발 환경에 따라 다르게 설정을 할 수 있다.
# Maven 도 Profile에 따라 설정을 달리 할 수 있다. 다만 표현력에 한계가 있다.

3. Incremental Build ( 증분 빌드 )
Gradle은 Incremental Build ( 증분 빌드 ) 를 지원하기 때문에, 변경된 Task만 다시 수행한다.
빌드 Task 단위 중에서 입력과 출력이 변경되지 않는 Task는 실행하지 않는다.

빌드 Task 단위는 compileJava , processResources , test , jar 등이 있다.
Task 단위는 빌드 스크립트 ( build.gradle ) 에서 관리한다.

# Maven과 Gradle은 수정된 Java 파일만 다시 컴파일하기 때문에 빌드 시간을 단축시킬 수 있다.
# 그럼에도 Gradle 이 Maven 보다 빠른 이유는 Build Cache , 병렬 빌드 , 증분 빌드 등이 합쳐져 있기 때문이다.

4. Build Cache
이전 빌드 결과를 로컬이나 원격 캐시에 저장해두었다가,
Task의 입력 ( Input ) 이 동일하다면 Task를 다시 수행하지 않고 캐시에서 출력 ( Output ) 을 가져와 재활용 한다.

5. Parallel Build ( 병렬 빌드 )
만약 여러 모듈을 빌드할 경우 Gradle은 멀티 코어 CPU 를 활용하여 동시에 빌드를 수행한다.
병렬 빌드를 수행하기 위해서 --parallel 옵션을 사용해야 한다.

6. 더욱 간단한 멀티 프로젝트 관리 ( 등록 include 과 의존성 implementation project )
여러 모듈을 관리할 때 include 명령으로 하위 프로젝트를 묶을 수 있다.
settings.gradle 파일에서 include "모듈명" 와 같이 작성하면, gradle이 해당 모듈도 함께 관리한다.

gradle에서 include를 통해 여러 모듈을 하나의 gradle 빌드에 등록하면,
루트 프로젝트에서 subprojects 설정을 통해 공통 빌드 설정을 하위 모듈이 공유할 수 있으며, 함께 빌드할 수 있고,
필요한 경우 implementation project(":모듈명")을 통해 다른 모듈의 코드를 참조할 수 있다.
그리고 IDE 에서 하나의 프로젝트 단위로 인식되어 리펙토링이나 코드 이동 및 자동 완성이 편리해진다.
즉, 여러 개의 모듈을 하나의 빌드 시스템으로 관리하도록 등록하는 것을 말한다.
# 공통 빌드 설정은 subprojects 로 설정한다.
# 아래와 같이 루트 프로젝트에서 저장소와 자바 버전을 등록하면, 하위 모듈이 모두 공유하게 된다.
subprojects {
    repositories {
        mavenCentral()
    }
    java {
        sourceCompatibility = JavaVersion.VERSION_21
    }
}

build.gradle 설정 파일의 dependencies 에서 implementation project(":common") 를 사용하여 다른 모듈을 의존성으로 등록하여 사용할 수 있다.
그렇게 되면 선언한 모듈은 common 을 의존하게 되고, 빌드 시 common 모듈이 먼저 빌드 되고, 해당 모듈이 빌드된다.
의존성 등록을 사용하기 위해선 먼저 include 를 통해 모듈을 등록해주어야 한다.

즉, include 는 여러 하위 모듈들을 하나의 멀티 프로젝트 빌드에 등록하는 명령어이고,
implementation project 는 등록된 하위 모듈을 현재 모듈의 의존성으로 추가하는 명령어이다.

Maven 도 하위 프로젝트를 묶고 관리할 수 있지만, 설정이 복잡하다.
# Gradle 에서 모듈이란 하나의 프로젝트를 여러 개의 하위 프로젝트 ( sub Project ) 로 나눈 것을 말한다.

7. Configuration Cache
Gradle의 Configuration 빌드 단계의 결과를 캐시에 저장하여,
이전과 동일한 빌드 설정 ( build.gradle ) 이라면 Configuration 단계를 다시 수행하지 않고, 캐싱된 결과를 재활용한다.
물론 빌드 설정에 영향을 주는 모든 입력이 동일 해야 한다. ( ex. Gradle 버전 , 빌드 옵션 , 플러그인 , setting.build , setting.build 등.. )

이를 통해 반복적인 빌드를 수행하여도 높은 성능을 낼 수 있다.

# 빌드 설정과 빌드 스크립트 ( build.gradle ) 는 동일하다.

[ Gradle의 빌드 단계 ]
1. Initialization : 어떤 프로젝트와 모듈을 빌드할지 결정하는 단계이다.
2. Configuration : 빌드 스크립트 ( build.gradle ) 를 읽어 어떤 Task 를 수행할지 결정한다.
이때 dependency 의존성을 해석하고, 어떤 Task 가 있고, Task 간의 의존 관계를 고려하려 결정한다.
Configuration Cache 를 통해 빌드 스크립트가 동일하다면 해당 단계를 생략하고 캐시된 결과를 재활용한다.
3. Execution : Configuration 단계에서 수행 결정된 Task 를 수행한다. ( compileJava > ProcessResources > classes > jar 등.. )
    ㄴ Task 수행 시 Incremental Build ( 증분 빌드 ) 를 통해 변경된 Task만 수행된다. ( Task의 입출력이 비교하여 UP-TO-DATE 일 경우 생략한다. )
    ㄴ 예를 들어 compileJava 단계에서 Java 소스, 컴파일 옵션, JDK 버전, 의존성이 동일하고 .class 파일이 존재하면 생략한다. ( Task :compileJava UP-TO-DATE )
    ㄴ 이때 Build Cache 를 통해 입력이 동일 할 경우 캐시에서 출력을 가져와 재활용한다.
    ㄴ 예를 들어 compileJava 단계에서 Java 소스, 컴파일 옵션, JDK 버전 , 의존성이 동일하면 Task를 수행하지 않고 캐시에서 .class 파일을 가져온다. (FROM-CACHE)
