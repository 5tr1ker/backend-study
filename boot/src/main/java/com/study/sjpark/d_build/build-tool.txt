[ Build-Tool ]
빌드 툴은 프로젝트를 빌드하는데 필요한 모든 작업을 자동화 하는 도구이며, 빌드 자동화말고도 그 외 프로젝트 작업도 자동으로 해준다.
Spring 에서는 빌드 툴을 Maven 과 Gradle 을 가장 많이 사용한다.

빌드 툴의 빌드 과정을 상세하게 나열하면 다음과 같다.
1. 라이브러리 다운로드
2. .java 소스 코드를 .class 파일로 컴파일
3. 테스트 코드 컴파일 및 실행
4. JAR/WAR 패키징
5. 실행
이와 같은 작업을 빌드 툴이 자동으로 수행해준다.
만약 빌드 툴이 없다면 수많은 라이브러리들을 직접 다운로드 하고, java 소스들을 class 파일로 컴파일하고 jar 로 묶은 다음 실행해주어야 한다.
프로젝트가 커질수록 라이브러리 관리와 빌드 과정이 많이 복잡해지는데, 빌드 툴은 이러한 과정을 설정 파일 하나로 자동화 해준다.

- 의존성 관리 및 라이브러리 다운로드
개발자는 필요한 라이브러리를 설정 파일에 기입하면, 빌드 툴은 해당 라이브러리들을 지정된 저장소에서 다운로드한다.
이때 하나의 의존성을 추가해도, 해당 라이브러리가 의존하는 관련 라이브러리가 전이 의존성에 의해 함께 다운로드가 될 수 있다.
이렇게 다운로드된 라이브러리들은 로컬 캐시 ( .gradle/caches ) 에 저장되며, 이미 캐시에 존재하는 라이브러리는 다시 다운로드 하지 않는다.

이후 다운로드된 라이브러리들을 포함하여 클래스패스를 구성하며, javac는 해당 클래스패스를 참조하여 필요한 외부 클래스를 해석하여 컴파일을 수행한다.

만약 빌드 툴이 없었다면 개발자는 필요한 라이브러리를 직접 찾아야 하고, 버전 충돌도 직접 해결해야한다.
또한 라이브러리의 전이 의존성을 직접 관리해야하며 컴파일 시 javac 가 참조할 클래스 패스를 직접 구성해야 한다.

# 의존성이란 프로젝트가 동작하기 위해 필요한 외부 라이브러리를 말한다.
# 클래스패스란 JVM이나 javac 가 필요한 라이브러리나 클래스 파일을 찾기 위해 탐색하는 경로 ( 라이브러리 및 클래스 파일이 어디있는지 설정 )

클래스패스를 구성한다는 것은 javac 가 참조해야 할 위치를 구성하는 것을 말한다.
클래스패스는 다음과 같이 구성할 수 있으며, 만약 라이브러리가 100개일 경우 해당 라이브러리들을 직접 기입해주어야 한다.
javac \
-cp lib/spring-core.jar:lib/spring-context.jar:lib/lombok.jar \
-d out \
src/Main.java

- 컴파일
빌드 툴은 내부적으로 javac 를 활용하여 프로젝트의 소스 코드를 .class 파일로 컴파일한다.
이때 클래스패스를 구성한 다음 javac를 실행하여 모든 소스 코드를 컴파일 한다.

# Gradle 에서는 gradle build, Maven 에서는 mvn compile 명령을 통해 javac 를 활용하여 프로젝트를 컴파일한다.

만약 빌드 툴이 없었다면 개발자는 컴파일 대상 소스 파일과 javac가 참조할 클래스패스를 직접 구성한 후 javac 명령을 직접 실행해야 한다.
javac \
-cp lib/spring-core.jar:lib/spring-context.jar:lib/lombok.jar \
-d out \
src/Main.java

- 테스트 실행
빌드 툴은 컴파일이 완료된 후 테스트 코드를 컴파일하고, JUnit 과 같은 테스트 프레임워크를 활용하여 테스트를 수행한다.
Gradle 에서는 gradle test , Maven에서는 mvn test 명령을 실행하면
빌드 툴은 src/test/java 디렉터리를 탐색하여 이름에 Test 가 포함된 클래스 파일들을 찾게 된다.
이후 JUnit Platform 을 실행하여 테스트 클래스 내부에 @Test 어노테이션이 붙은 메서드들을 순차적으로 실행하게 된다.

만약 1개 이상의 테스트가 실패할 경우 빌드는 실패하게 되며, 그 이후 단계인 JAR/WAR 패키징 단계는 수행되지 않는다.

# 테스트 패턴 ( 테스트 파일 이름 규칙 및 패키지 규칙 ) 은 다른 패턴으로 변경할 수 있지만, 보통 기본 값을 사용한다.

# 테스트 관련 라이브러리는 JUnit 5 가 가장 많이 사용되며, Spring boot 에서는 거의 표준으로 사용된다.
# 그래서 spring-boot-starter-test 라이브러리를 다운로드 하면, JUnit 5 가 다운로드 된다.

- JAR/WAR 패키징
컴파일 이후 .class 파일들을 하나의 압축된 실행 단위로 묶는 것을 패키징이라 한다.
Gradle에서는 gradle bootJar, Maven 에서는 mvn package 명령을 수행하면, 패키징을 수행한다.

# JAR ( Java Archive ) 는 여러 개의 class 파일과 리소스 파일을 하나로 묶은 압축 파일이다. 확장자가 jar 인 zip 이라고 보면 된다.
# WAR ( Web Archive ) 는 웹 어플리케이션을 위한 패키지로 Tomcat 이나 Jetty 와 같은 WAS 에 배포할 때 활용된다.

일반 Java 프로젝트에서는 gradle jar 를 사용하지만, Spring Boot에서는 gradle bootJar 를 사용한다.
bootJar는 모든 클래스 파일과 라이브러리, 내장 Tomcat 을 함께 패키징 해주기 때문에 [ "java -jar" ] 명령어를 통해서 서버를 바로 실행할 수 있다.
이러한 jar는 "실행 가능한 JAR ( Executable Jar )" 라고 한다.

반면 [ "gradle jar" ] 로 패키징할 경우 클래스 파일만 패키징한다.

Maven의 경우 mvn package 를 통해 jar 혹은 war가 생성되며 ( 설정에 따라 다름 ),
Spring boot 환경일 경우 Spring Boot Maven Plugin 를 활용하여 실행 가능한 JAR 를 만들 수 있다.

만약 빌드 툴이 없었다면 개발자는 컴파일된 클래스 파일들과 리소스들을 모아 jar 로 직접 패키징해야 하며,
Manifest 파일과 포함할 리소스도 직접 관리해야 한다.

컴파일 시점에 포함되는 파일들은 다음 것들이 있다.
src/main/java  > 소스 파일들로 컴파일된 .class 파일들
src/main/resources  > 리소스들 보통 그대로 들어감 ( 이미지 , yml , xml 등.. )
META-INF/MANIFEST.MF > JAR 파일에 대한 메타데이터 ( 실행에 필요한 정보 ( 메인 클래스 ) 와 버전 등 메타 정보를 가지고 있으며,
                        Spring Boot 의 실행 가능한 JAR 의 경우 시작 클래스 정보도 포함된다 )

- 실행
패키징이 완료되어 생성된 JAR 혹은 WAR 파일을 실행하여 어플리케이션을 구동할 수 있다.
Spring Boot 프로젝트에서 생성된 실행 가능한 JAR 파일은 "java -jar" 명령으로 실행 시킬 수 있다.

개발 환경에서는 gradle bootRun 혹은 mvn spring-boot:run 명령어를 통해 패키징 과정 없이 프로젝트를 바로 실행 시킬 수 있다.
해당 명령은 테스트 및 패키징 단계를 거치지 않아 개발 환경에서 어플리케이션을 빠르게 실행시킬 수 있다.

[ 프로젝트 작업 자동화 ]
빌드 툴의 핵심은 빌드 자동화이며, 빌드 과정에서 사용할 수 있는 다양한 작업을 제공한다.

- 빌드 삭제 ( clean )
이전 빌드의 결과물을 삭제한다.
보통 build/ 디렉터리를 삭제하는 작업이며, 이전 빌드 결과물이 남아 발생할 수 있는 문제를 방지하기 위해 사용한다.

- 배포 ( publish )
빌드가 된 결과물을 다른 곳으로 배포하는 작업이다.
이때 빌드가 완료된 결과물 ( JAR, WAR 등 )을 외부 저장소나 서버에 배포하는 작업이다.

- 테스트 커버리지 생성 ( jacocoTestReport )
테스트 커버리지는 테스트 코드가 실제 어플리케이션 코드를 얼마나 실행했는지를 나타내는 지표이다.

JaCoCo 플러그인이 적용된 상태에서 gradlew test 명령과 gradlew jacocoTestReport 명령을 사용하면,
JaCoCo ( Java Code Coverage ) 라이브러리를 활용하여 테스트 커버리지를 측정할 수 있다.
# JaCoCo 는 Java 어플리케이션의 테스트 커버리지를 측정하는 라이브러리이다.

gradlew test 명령으로 테스트를 수행하면, JaCoCo 플러그인이 테스트 실행 데이터를 수집한다.
그 후 gradlew jacocoTestReport 명령을 사용하면 수집한 데이터를 기반으로 리포트를 생성한다.
이때 리포트에는 클래스, 메서드, 코드 라인 등의 커버리지 정보가 기록되어 있다.
Class          Line Coverage
UserService         95%
OrderService        82%
PaymentService      60%

하지만 테스트 커버리지가 높다고 좋은 테스트라고 볼 수 없다.
좋은 테스트는 테스트 품질이 더욱 중요하고, 테스트 커버리지는 참고 지표로 사용한다.
> 테스트 품질 : 테스트가 요구사항을 올바르게 검증하였는가?
> 테스트 커버리지 : 테스트가 실제 코드를 얼마나 실행하였는가?
