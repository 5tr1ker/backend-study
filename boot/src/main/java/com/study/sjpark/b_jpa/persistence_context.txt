[ 영속성 컨텍스트란 ]
영속성 컨텍스트는 엔티티를 관리하는 1차 캐시 공간으로 Database를 객체 지향적으로 다루게 해주는 실행 환경이다.
뿐만 아니라 변경 감지와 동일성 보장, 쓰기 지연 , 1차 캐시 , 지연 로딩의 이점을 가지고 있다.

변경 감지 : 엔티티의 최초 상태를 스냅샷에 저장한 후 트랜잭션이 종료될 때 현재 상태와 비교하여 변경된 부분에 대해서 자동으로 update sql 을 생성한다.
동일성 보장 : 같은 식별자를 가진 엔티티는 동일한 객체를 반환하여 동일성을 보장한다.
쓰기 지연 : 데이터 변경 및 추가 쿼리를 쓰기 지연 저장소에 보관하고 있다가 commit / flush 시 DB에 한꺼번에 반영한다.
1차 캐시 : 데이터 조회 시 DB를 바로 조회하지 않고, 1차 캐시를 통해 먼저 조회한 후 객체가 없을 경우 DB를 조회한다.
지연 로딩 : 연관 관계의 엔티티를 미리 가져오지 않고 사용할 때 프록시를 통해 데이터를 가져온다.

[ 엔티티 생명주기 ]

- 비영속
단순 객체로 영속성 컨텍스트와 연관이 전혀 없는 객체이다.

- 영속
영속성 컨텍스트에 의해 관리되고 있는 객체이다. 해당 객체는 1차 캐시, 변경 감지, 동일성 보장 등이 적용되는 객체이다.
해당 상태이여야 JPA에 관리를 받아 DB에 적재되거나 수정된다.

- 준영속
영속성 컨텍스트에서 분리된 상태로 관리가 더이상 되지 않는 객체이다.
해당 객체는 변경 감지가 되지 않는다.

- 삭제
삭제 대상인 엔티티이다. commit이 되었을 경우 DB 에서 DELETE 되는 대상이다.

[ flush 와 commit ]

- flush
flush는 영속성 컨텍스트의 변경 내용을 DB에 반영하는 시점이다.
INSERT / UPDATE / DELETE 쿼리가 실행되지만, 트랜잭션은 아직 유지되고 있기 때문에 rollback을 할 수 있다.

flush 는 트랜잭션 commit 시, JPQL 실행 전, 개발자가 직접 호출할 때에만 실행된다.
JPQL 실행 전에 flush가 되는 이유는 DB와 영속성 컨텍스트의 상태를 맞추기 위해서이다.

flush를 너무 자주 사용할 경우 DB에 부하를 줄 수 있기 때문에 필요할 때에만 활용해야 한다.

- commit
commit은 flush를 수행하고, 트랜잭션을 종료하여 DB에 최종 반영을 한다.
이는 트랜잭션이 종료되기 때문에 rollback을 할 수 없다.

[ 연관관계 주인 ]
두 엔티티가 서로 연관 관계에 있을 때 외래키를 관리할 주인을 정해야 한다.
연관관계 주인만이 외래키를 변경할 수 있으며, 주인이 아닌 쪽에는 mappedBy 를 지정하여 읽기 전용으로 설정해야 한다.

이때 연관관계 주인이 아닌 곳에서 연관 관계 객체를 수정한다고 해도 DB에 반영되지 않는다.

[ N + 1 이슈 ]
조회한 데이터의 연관 엔티티를 조회하는 과정에서 추가 쿼리가 N번 발생하는 현상이다.
예를 들어 연관 데이터가 20개가 있을 경우 20번의 SELECT 조회가 추가로 실행되는 현상을 말한다.

해당 현상은 지연 로딩 및 즉시 로딩 모두 발생할 수 있다.
지연 로딩의 경우 연관 관계 엔티티의 프록시가 초기화 될 때 마다 추가 쿼리가 발생하며,
즉시 로딩는 JPQL을 통해 조회할 경우, 해당 연관관계의 엔티티의 갯수만큼 추가 SELECT 쿼리가 발생한다.

해당 현상을 해결하기 위해서는 연관 관계의 로딩 전략을 LAZY로 설정한 다음 fetch join으로 조회하여 한번에 조회해야 한다.
그 뿐만이 아니라 @EntityGraph를 사용하거나 @BatchSize 를 사용하는 방법으로도 N + 1 문제를 해결할 수 있다.

# 즉시 로딩으로 설정해도 find() 메서드로 조회할 경우 JOIN 을 통해 연관 관계의 엔티티도 함께 가져온다.

- fetch Join

@Query("select m from Member m join fetch m.orders")

JPQL에서 연관 엔티티를 JOIN으로 한번에 가져오는 방법이다.
해당 방법은 DB에서 JOIN을 통해서 한번에 가져오는 것과 동일하며, 단 한번으로 연관 엔티티의 데이터도 함께 가져올 수 있어 가장 직관적이다.
하지만 1:N 구조일 때 중복 데이터가 발생할 수 있어 쿼리에서 중복 제거문을 추가한다. ( 회원1:주문5 일 경우 결과 rows가 5개이며, 이는 같은 회원 데이터가 5개가 들어간다. )
그리고 Pageable 과 fetch Join 는 함께 사용하지 못하기 때문에 페이징이 안된다.

fetch join은 데이터가 많지 않고, 조회 성능이 중요한 API에서 사용하면 된다.

- @EntityGraph

@EntityGraph(attributePaths = {"orders"})
List<Member> findAll();

fetch Join을 어노테이션으로 선언하는 방식이다.
이는 내부적으로 fetch Join 과 동일한 방식으로 동작하며, JPQL을 사용하지 않아도 JOIN으로 한번에 가져온다.
Spring Data JPA 에서 사용하기 편리하며 코드가 깔끔해지는 장점을 가지고 있지만, 복잡하고 세밀한 쿼리를 작성하는데 한계가 있다.

EntityGraph는 Repository 코드를 깔끔하게 유지하고, 간단한 조회를 해야할 때 사용한다.

- @BatchSize
BatchSize는 N+1 을 완화시키기 위한 전략으로, 설정한 크기만큼 IN 쿼리를 활용하여 한번에 조회한다.
예를 들어 연관 엔티티가 50개이고 BatchSize를 10 으로 설정했다면, 하나의 IN절에 10개의 키값을 넣어 총 5번의 SELECT 문을 호출한다.

BatchSize는 페이징이 가능하다는 장점을 가지고 있지만, 쿼리가 여러번 나간다는 단점과 size를 잘못 잡으면 성능이 저하되는 단점을 가지고 있다.
BatchSize는 페이징을 꼭 사용해야 하며, 컬렉션 fetch join을 2개이상 사용해야 할 때 사용한다.

- 일반 fetch join과 컬렉션 fetch join
fetch join은 연관 관계에 따라서 1:1인 경우 일반 fetch join, 1:N인 경우 컬렉션 fetch join으로 나뉜다.
일반 fetch join인 경우 페이징을 사용할 수 있고 다중 fetch join을 사용할 수 있지만, 컬렉션 fetch join은 그러지 못한다.

- 컬렉션 fetch join과 페이징을 함께 사용하지 못하는 이유
컬렉션 fetch join을 사용할 경우 1:N 구조에서 중복된 데이터가 발생한다. limit는 조회된 rows 수를 기준으로 데이터를 자르기 때문에 중복된 데이터로 인해
데이터 왜곡이 발생할 수 있다.

예를 들어 회원 한명당 주문이 3건씩 있다고 가정할 경우 아래와 같은 데이터가 추출될 것이다.
회원1:주문1
회원1:주문2
회원1:주문3
회원2:주문1
회원2:주문2
회원2:주문3
여기서 limit 2 를 사용할 경우, 위와 같은 데이터가 조회될 것이라 기대할 수 있지만 실제로는 아래와 같은 데이터가 추출된다.
회원1:주문1
회원1:주문2
따라서 JPA는 이러한 문제 때문에 fetch Join을 사용할 경우 페이징을 사용하지 못하게 막았다.

- 컬렉션 fetch join 2번 이상 사용 금지
컬렉션 fetch join 이란, 연관관계의 엔티티가 다수인 것을 말한다. 즉, OneToOne 이 아닌 OneToMany를 말한다.
JPA에서는 아래와 같이 컬렉션 fetch join 을 2번 이상 사용하는 것을 금지하고 있다.
select m from Member m
join fetch m.orders
join fetch m.coupons
그 이유는 fetch join은 곱연산이기 때문에 Member x orders x coupons 의 수 만큼 rows가 생성되기 때문에 메모리에 부하를 일으킨다.
이는 어플리케이션의 성능이 저하되고 심각한 경우 OOM이 발생할 수 있는 위험성 때문에 JPA에서는 금지하고 있다.

따라서 연관관계 중에서 컬렉션이 많은 경우 하나만 fetch Join을 사용하고, 나머지는 BatchSize 를 사용하기도 한다.
