[ MongoDB 란? ]
MongoDB는 문서 지향 데이터 모델 ( Document DB ) 를 사용하는 데이터베이스로 NoSQL 중에서 가장 많이 사용된다.
문서 지향 데이터 모델을 활용하면 정형 및 비정형 데이터를 빠르고 쉽게 통합할 수 있다.

MongoDB ( NoSQL ) 은 초당 동시 처리가 중요한 업무 및 로그나 이력 등 단순한 기록형 업무에 적합하다.
반 정규화를 기반으로 설계되며, 비정형 스키마 구조로 미리 스키마를 선언하지 않는다.
또한 NoSQL 은 쿼리 외에 다양한 API를 통해 데이터를 적재 및 조회할 수 있으며, 쿼리 프로세싱이 단순화 되어 있어 대용량 데이터를 처리하는데 성능이 우수하다.
하지만 데이터 중복으로 인해 데이터의 일관성이 떨어지고 용량이 늘어난다는 단점을 가지고 있다.

[ MongoDB 만의 장점과 특징 ]

1. 고가용성 ( Reliability )
MongoDB는 기본적으로 1개의 Primary 서버와 2개의 Secondary 서버로 구성된 ReplicaSet 구조를 지원하여 Primary 서버에 장애가 발생하여도 서비스를 지속적으로 제공할 수 있다.
Secondary 서버는 주기적으로 Primary 서버의 데이터를 복제하여 동일한 상태를 유지하고 있다가, Primary 서버에 장애가 발생하면
하나의 Secondary 서버가 Primary 서버로 전환되어 서비스가 계속해서 유지할 수 있게 해준다.

2. 확장성 ( Scalability )
MongoDB는 데이터와 트래픽 증가로 인한 수평확장 ( scale-out ) 과 데이터 샤딩을 통한 수평 확장이 가능하다.

MongoDB는 특정 샤드 에서 다른 샤드로의 데이터 밸런싱을 서비스 중단 없이 진행할 수 있다.
데이터 밸런싱을 통해 모든 샤드가 균등한 데이터를 가지게 되며, 특정 샤드에 데이터가 몰리는 현상을 방지할 수 있다.

추가로 단일 Replica Set 에서 샤드로의 전환을 서비스 중단 없이 진행할 수 있으며, 샤드의 축소 및 확장도 온라인으로 가능하다.

3. 유연성 ( Flexibility )
요구 사항 변경에 따라 데이터의 구조가 변경이 되어도, 구조 변경 작업 없이 데이터를 저장해주기만 하면 된다.

RDBMS의 경우 데이터 변경에 따라 스키마를 변경하거나 연관 관계를 맺어주어야 하지만, MongoDB는 그저 데이터를 저장만 하면 된다.
MongoDB는 Json 기반의 Document 모델을 사용하기 때문에 다양한 형태의 데이터를 한번에 담을 수 있고, 여러 가지 정보를 한눈에 볼 수 있다.
그에 반에 RDBMS는 나뉘어진 테이블 간 연관 관계로 맺어져 있어 데이터를 한눈에 보기 어렵다.

4. 인덱스 지원 ( Index )
대부분의 NoSQL과는 다르게 MongoDB는 인덱스를 지원한다.
이를 통해 특정 필드에 필요한 만큼 인덱스를 생성할 수 있으며, 대용량 데이터에서 다양한 조건으로 데이터를 빠르게 조회할 수 있다.
MongoDB는 다음과 같이 다양한 인덱스를 제공한다.

Hashed Index : 샤드 클러스터에서 데이터를 균등하게 분배하고자 사용한다.
Multikey Index
Partial Index
TTL Index : 제한시간을 설정하여 일정 시간이 지나면 자동으로 삭제한다.
Geospatial Index : 공간 내 거리나 범위를 계산한다.

5. 고성능

[ MongoDB 모델 구성 ]
MongoDB는 Server 와 Database , Collection , Document 와 Fields 로 구성되어 있다.

- Server : 여러 개의 Database 를 가지고 있는 서버

- Database : Collection 의 물리적인 컨테이너

- Collection :
RDBMS의 Table 과 비슷한 개념이지만, 동적 스키마로 스키마를 정의하진 않는다. ( Schema-less )
또한 Document를 여러개 가지고 있으며, Document의 내부에 위치한다.

- Document :
RDBMS에서 row 와 비슷하며, 한개 이상의 key-value 쌍을 이룬다.
각각의 데이터들은 key 와 fields 로 구성되어 있으며, key는 컬럼 명, fields 는 저장 값이 된다.
여기서 Document 내부에 있는 배열 타입의 fields 는 SubDocument 라고 한다.

{
  _id: POST_ID, title: POST_TITLE,
  content: POST_CONTENT,
  author: POST_WRITER,
  hashtags: [ TAG1, TAG2, TAG3 ], // subdocument
  time: POST_TIME
  comments: [ // subdocument
     {
       username: COMMENT_WRITER,
       mesage: COMMENT_MESSAGE,
       time: COMMENT_TIME
     },
     {
       username: COMMENT_WRITER,
       mesage: COMMENT_MESSAGE,
       time: COMMENT_TIME
     }
  ]
}

[ MongoDB의 JSON / BSON ]
MongoDB 는 데이터를 BSON 형태로 저장하고 관리하고 있지만, 사용자에게 보여주거나 요청을 보낼 때에는 Json 문법을 활용한다.
JSON는 직관적이고 이해하기 쉽다는 장점을 가지고 있지만, 텍스트 기반으로 구문 분석이 느리고 공간 효율성이 낮은 단점을 가지고 있다.

그렇기 때문에 JSON 구조의 장점을 그대로 가져가고, 기계가 빠르게 읽을 수 있는 Binary 형태인 BSON 을 활용하여 데이터를 적재 및 관리한다.
따라서 사용자에게 보여질 때에는 JSON의 형태로 보여주고, 저장 및 네트워크 전송 시에는 BSON 형태로 관리한다.

[ MongoDB의 컬렉션간 관계 ]
MongoDB에서 컬렉션간 연관 관계를 맺는 방법에는 크게 2가지 종류가 있다.
1:1 관계인 경우 하나의 Document 에다가 모두 입력하면 되지만, 1:N 관계 혹은 M:N 관계인 경우 아래와 같은 방법을 활용할 수 있다.

- Document - Embedded
한 Document에 연관되어 있는 Document를 Embedded Document로 관리하는 것을 말한다.
즉, Document 안에 연관된 Document를 통째로 넣어서 관리하는 것을 말한다.
{
    "name" : "sjpark",
    "age" : 26,
    "address" : {
        "street" : "123123",
        "city" : "seoul",
        "zip" : "12345",
        ...
    }
}

가장 단순한 방법이지만, 데이터 중복이 발생하여 Document의 크기가 커진다는 단점을 가지고 있다.

- Document - Reference
연관된 Document 의 _id 값을 저장하는 방법으로, MySQL의 외래키와 비슷한 개념이다.
{
    "name" : "sjpark",
    "age" : 26,
    "address" : "1bj41q0281o32", <- Address Document 의 _id 값
    "car" : ["2v5r1539jga" , "dg414r1gaa"] <- 만약 M:N 구조일 경우 배열로 표현할 수 있다.
}
