[ EventLoop 에서의 Blocking ]
Spring WebFlux 환경에서 블로킹 작업을 잘못 다룰 경우 CPU 자원은 남아있는데 응답이 지연되는 현상이 발생할 수 있다.
Webflux에서는 쓰레드 구조가 크게 이벤트 루프와 워커 쓰레드로 나뉘는데, 블로킹 작업을 이벤트 루프에서 처리할 경우
Spring Webflux의 장점을 잘 활용하지 못할 수 있다.

해당 문서는 Spring webflux 환경에서 블로킹 작업을 효율적으로 잘 다루는 방법을 정리할 것이다.

[ 워커 쓰레드를 늘리는 것이 아닌 이벤트 루프 쓰레드를 늘리는 방법? ]
이벤트 루프는 기본적으로 ( CPU 코어 수 * 2 ) 개이며, 워커 쓰레드는 기본적으로 ( CPU 코어 수 * 10 ) 개이다.
필요에 따라 이벤트 루프와 워커 쓰레드는 쓰레드 수를 늘릴 수 있다.
하지만 보통 이벤트 루프를 늘리는 것 보다 워커 쓰레드를 늘리는 것을 추천한다,

그 이유는 이벤트 루프는 CPU-Bound 특성이기 때문에 이벤트 루프를 늘릴 경우 오히려 성능이 저하될 수 있다.
이벤트 루프를 늘리면 컨텍스트 스위칭의 빈도가 높아지고 CPU Cache Miss가 발생할 확률이 높아진다.

반대로 워커 쓰레드는 I/O-Bound 특성이기 때문에 쓰레드의 수가 늘어난 다고 해서 성능이 크게 저하되지는 않는다.
그렇기 때문에 네트워크 작업, DB 작업 등과 같이 Blocking이 되는 로직을 해당 쓰레드에서 처리하는 것이 좋다.

그렇기 때문에 이벤트 루프에서는 짧고 빠른 작업만 주로 수행하는 것이 좋고, 워커 쓰레드에서는 Blocking 이 되는 로직을
처리하는 것이 좋다.

- 워커 쓰레드와 이벤트 루프
이벤트 루프는 기본적으로 1개의 이벤트 루프당 1개의 쓰레드를 가지며, 각각의 이벤트 루프 쓰레드는 독립적이다.
즉, 이벤트 루프는 서로 간섭하지 않는다. 그 이유는 컨텍스트 스위칭과 락을 최소화 하여 최대한의 성능을 내기 위해서이다.

이벤트 루프는 새로운 HTTP Connection 이 들어올 때 마다 특정 이벤트 루프에 배정이 되며,
해당 Connection 으로 들어오는 모든 요청은 동일한 이벤트 루프에서 처리하게 된다. 그리고 Connection 이 종료될 때 까지
다른 이벤트 루프로 변경되지 않는다. 그렇기 때문에 이벤트 루프의 부하가 균등하지 않을 수 있다.
Connection 과 이벤트 루프는 단위 고정이기 때문에 특정 Connection에서 느린 요청이 몰리면 해당 Connection 의 전체 처리 속도가
저하될 수 있다.
그에 반에 워커 쓰레드는 특정 이벤트 루프에 종속되어 있지 않다. 여러 이벤트 루프가 공용해서 사용한다.

한 이벤트 루프는 하나의 쓰레드가 전담하여 작업들을 순차적으로 처리한다.
반면 워커 쓰레드는 여러 쓰레드가 OS 스케쥴링으로 CPU 실행 기회을 받아야 작업을 처리하기 때문에 성능의 차이가 발생한다.
# 물론 이벤트 루프는 순차 처리 기반이고, 워커 쓰레드는 병렬 처리 기반이기 때문에 상황에 따라 다르다.
또한 워커 쓰레드의 개수가 지나치게 많으면 컨텍스트 스위칭 비용 증가로 인해 오히려 성능이 저하된다.

이러한 특성으로 이벤트 루프에서 블로킹이 발생하면 뒤에 작업들이 모두 지연되어 해당 이벤트 루프의 처리 능력은 멈춘다.
하지만 워커 쓰레드는 블로킹 상태여도 CPU 실행 기회를 다른 쓰레드에게 부여되기 때문에 이벤트 루프보다는 덜 치명적이다.
물론 워커 쓰레드의 대부분이 블로킹 상태가 되면 처리 가능한 쓰레드가 고갈되어 큐가 밀리거나 응답 시간이 증가하게 될 것이다.

- 모든 작업을 무조건 워커 쓰레드에 넘기면 안되는 이유
모든 작업을 워커 쓰레드에 넘기면 오히려 성능이 저하될 수 있다.
그 이유는 이벤트 루프가 워커 쓰레드로 작업을 넘길 경우 최소 2번의 컨텍스트 스위칭 비용이 발생하며, 서로 다른 쓰레드로 인해
CPU cache locality 를 유지하지 못해 cache miss 가 증가하고, 메모리 접근 비용이 증가하여 성능이 저하된다.
또한 boundedElastic 기반의 워커는 내부적으로 작업 대기 큐를 가지고 있어 모든 작업을 큐로 넘길 경우
큐가 포화 상태가 되어 응답 대기 시간이 증가할 수 있다.

이러한 이유로 Blocking이 없는 짧은 작업의 경우 워커로 넘기는 비용보다 이벤트 루프에서 처리하는 비용이 더 적기 때문에
이벤트 루프에서 작업을 처리해서 반환하는 것이 좋다. 반대로 Blocking 작업이 있거나 긴 작업이 있는 경우 워커 쓰레드로 넘겨서
작업을 처리하는 것이 바람직하다.

[ EventLoop에서 병목이 발생한 것을 어떻게 알 수 있을까? ]
가장 간단한 방법으로는 쓰레드 덤프를 뜨는 것이다. 아래의 명령어를 통해서 쓰레드의 상태를 txt로 남길 수 있다. ( 윈도우 기준 )
jstack PID > thread.txt

쓰레드 덤프 내에서 이벤트 루프와 워커 쓰레드는 다음과 같은 이름을 가지고 있다.
reactor-http-nio-N ( 이벤트 루프의 이름이며 N 자리에는 쓰레드의 번호가 부여된다. CPU 코어가 4라면 8개의 이벤트 루프를 볼 것이다. )
boundedElastic-N ( 워커 쓰레드의 이름이며 N 자리에는 워커 쓰레드의 번호가 부여된다. )
# 기본적으로 이벤트 루프는 CPU 코어 x 2 개가 기본 값이다.
# 워커 쓰레드는 기본 CPU 코어 x 10 개가 기본 값이다.

- EventLoop의 병목 발생 시
만약 이벤트 루프에서 블로킹으로 인한 병목이 발생을 했을 경우 쓰레드 덤프에는 아래와 같은 로그가 남게 된다.
"reactor-http-nio-2" #43 ... 생략
   java.lang.Thread.State: TIMED_WAITING (sleeping)
	at java.lang.Thread.sleepNanos0(java.base@24.0.2/Native Method)
	at java.lang.Thread.sleepNanos(java.base@24.0.2/Thread.java:482)
	at java.lang.Thread.sleep(java.base@24.0.2/Thread.java:513)
... 생략

실제로 모든 이벤트 루프의 쓰레드가 위와 같은 상태를 가지고 있을 것이다.
위의 로그에는 Thread.sleep() 으로 인한 블로킹으로 쓰레드가 멈춰 있는 상태 ( TIMED_WAITING (sleeping) ) 를 나타내고 있다.
즉, 모든 이벤트 루프가 블로킹이 되어 더이상 작업을 하고 있지 않은 상태이다.
실제로 테스트 시 Thread.sleep() 메서드를 활용해서 1.5초간 병목을 주었다.

쓰레드 덤프에는 위와 같이 어떤 메서드로 인한 블로킹이 발생했는지 나타내기 때문에 문제가 되는 로직을 어렵지 않게 개선할 수 있을 것이다.

- EventLoop 정상 상태 ( 워커 쓰레드 병목 )
반대로 EventLoop가 아닌 워커 쓰레드에게 블로킹 로직을 주었을 때 쓰레드 덤프는 아래와 같은 로그를 남기게 된다.
"reactor-http-nio-5" ... 생략
   java.lang.Thread.State: RUNNABLE
	at sun.nio.ch.WEPoll.wait(java.base@24.0.2/Native Method)
	at sun.nio.ch.WEPollSelectorImpl.doSelect(java.base@24.0.2/WEPollSelectorImpl.java:108)
... 생략

위의 상태 값은 I/O 이벤트를 기다리고 있는 상태로 실제로 일을 하지 않고 있는 상태이다.
신기하게 병목을 주고 있는 상태에서 8개의 모든 이벤트 루프는 위의 상태 값을 가지고 있었다.
그 이유는 병목 로직을 워커 쓰레드에게 위임하고 있었기 때문이다.

실제로 병목 상태에서 다수의 워커 쓰레드가 블로킹 상태였는데, 대부분의 워커 쓰레드가 아래와 같은 로그를 출력했다.
"boundedElastic-87" #138 [20456] ...생략
   java.lang.Thread.State: TIMED_WAITING (sleeping)
	at java.lang.Thread.sleepNanos0(java.base@24.0.2/Native Method)
	at java.lang.Thread.sleepNanos(java.base@24.0.2/Thread.java:482)
	at java.lang.Thread.sleep(java.base@24.0.2/Thread.java:513)
... 생략
로그를 보면 Thread.sleep() 블로킹 로직이 워커 쓰레드에서 진행되고 있었다.
EventLoop에서 병목이 발생하던 상황과는 다르게 워커 쓰레드에서 블로킹 로직이 실행되고 있었다.

[ 병목 테스트 ]
병목 테스트 시 주의 사항으로는 이벤트 루프 갯수 이상의 동시성으로 부하를 주어야 한다.
예를 들어 이벤트 루프의 쓰레드가 8개일 때, 동시성을 8 이하로 줄 경우 병목을 제데로 재현하지 못할 수 있다.
쉽게 말해 EventLoop에서 병목을 주든, 워커 쓰레드에서 병목을 주든 차이가 없을 것이란 것이다.
